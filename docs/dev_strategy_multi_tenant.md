# 다중 지자체·업체 대응 개발 전략

> 작성일: 2026-05-12  
> 대상 프로젝트: wci-mvp (폐기물 수거 정보 시스템)  
> 배경: 전국 지자체 및 위탁업체별 요구사항 차이로 인한 반복 수정 문제 해결

---

## 1. 핵심 원인 진단

현재 구조는 **요구사항 = 코드** 방식이다.  
지자체·업체가 새 요구사항을 낼 때마다 개발자가 코드를 수정해야 하는 구조적 의존이 반복 수정의 근본 원인이다.

---

## 2. 해결 방향 3가지

### 방향 1. 계층형 설정(Configuration) 시스템 — 단기

```
전국 공통(SUPER) → 지자체(MUNI) → 업체(CONTRACTOR) → 작업자(WORKER)
```

각 레벨이 하위 레벨에게 "어떤 폼 카드를 쓸지, 어떤 필드를 켤지"를 설정으로 내려보내는 구조.  
요소수 삭제, 공공봉투 삭제, 반입장소 설정 같은 요청이 전부 **설정값 변경**으로 흡수된다.

- **장점**: 기존 코드 최소 변경, 빠른 대응
- **단점**: 설정 항목이 늘수록 설정 UI 관리가 복잡해짐

### 방향 2. 폼 카드 모듈화 + 지자체 프로필 — 중기

폼을 **독립 카드 단위**(주유, 요소수, 공공봉투, 중량봉투, 대형폐기물 등)로 분리하고,  
지자체·업체에 "어떤 카드를 몇 번째 순서로 쓰는지" 프로필을 DB에 저장.  
온보딩 시 유사 지자체 프로필을 복사해서 빠르게 설정.

- **장점**: 지자체 온보딩 자동화
- **단점**: 카드 간 의존성이 생기면 복잡도 상승

### 방향 3. 저코드 폼 빌더 — 장기

관리자가 UI에서 드래그앤드롭으로 필드·테이블을 추가·삭제·순서변경.  
필드 유형 라이브러리(숫자입력, kg단위 테이블, 다중선택 등)를 미리 만들어두고 조립.  
방향 1·2가 커버하지 못하는 완전히 새로운 필드 유형 요구가 나왔을 때만 개발자 개입.

- **장점**: 개발자 의존 0%
- **단점**: 초기 구축 비용 높음, 데이터 집계·통계 설계 어려움

### 권장 순서

| 단계 | 방향 | 효과 |
|------|------|------|
| 단기 | 계층형 ON/OFF 설정 | 기존 코드 최소 변경 |
| 중기 | 카드 모듈 프로필 | 지자체 온보딩 자동화 |
| 장기 | 폼 빌더 | 개발자 의존 제거 |

---

## 3. 개발 단계별 실행 전략

### Phase 1. 흡수기 — "일단 다 받아라"

> 기간: 지금 ~ 첫 3개 업체 온보딩 완료  
> 목표: 요구사항 거절 없이 빠르게 수용, 패턴 수집

#### 전술 ①  routeDetail JSON 의도적 활용

`routeDetail` 텍스트 컬럼에 JSON을 자유롭게 쌓는 방식은 이 단계에서 강점이다.  
새 필드 요청이 오면 DB 스키마 변경 없이 JSON 키만 추가한다.  
정규화는 안정화 단계에서 한꺼번에 처리한다.

#### 전술 ②  요구사항 태그 기록

요청이 들어올 때마다 아래 형식으로 기록한다.

| 요청 내용 | 요청 업체 | 지자체 | 분류 |
|---------|---------|--------|------|
| 요소수 항목 삭제 | A업체 | 서울 | 필드 ON/OFF |
| 반입장소 선택 추가 | B업체 | 부산 | 관리자 설정 |
| 대형폐기물 테이블 | A·C업체 | 공통 | 신규 모듈 |
| 중량계 수거량 입력 | B업체 | 인천 | 신규 모듈 |

3개 업체만 쌓여도 공통 패턴이 보이기 시작한다.

---

### Phase 2. 관찰기 — "겹치는 것을 찾아라"

> 기간: 3~7개 업체 운영 중  
> 목표: 패턴 분류, 설정 테이블 도입, 온보딩 표준화

#### 요구사항 분류 기준

```
2개 이상 업체가 요청한 것  →  코어 기능으로 격상
1개 업체만 요청한 것       →  설정(ON/OFF)으로 처리
아직 아무도 안 쓰는 것     →  보류
```

#### 설정 테이블 도입

이 시점에 설정 테이블 하나를 만들어 ON/OFF를 흡수한다.  
코드는 그대로 두고 "이 업체는 요소수 카드 OFF" 같은 설정만 DB에 저장.

#### 표준 온보딩 체크리스트 작성

새 업체가 들어오면 담당자가 체크리스트 항목만 설정하면  
**2시간 안에 세팅 완료**되는 수준을 목표로 한다.

---

### Phase 3. 안정화 — "굳히기"

> 기간: 7~10개 업체 운영, 또는 1년 운영 후  
> 목표: DB 정규화, 코드 구조 정리, 기능 동결

#### 안정화 타이밍 신호 3가지

1. 새 업체가 들어와도 **기존 설정 조합으로 대부분 커버**됨
2. `routeDetail` JSON에 특정 키가 **전 업체 공통으로** 들어감
3. 고객 요청이 "새 기능"보다 **"편의성 개선"** 위주로 바뀜

#### 안정화 작업 3단계

**① JSON → 정규 컬럼 마이그레이션**  
`routeDetail`에 쌓인 공통 패턴을 실제 DB 컬럼으로 올린다.  
이후 집계·통계 쿼리가 가능해진다.

**② 설정 UI 완성**  
관리자가 코드·개발자 없이 업체 온보딩을 완료할 수 있는 어드민 화면.

**③ 기능 동결 + QA 기간 선언**  
한 분기 동안 신규 기능 없이 버그픽스와 성능만 잡는다.  
사용자들은 이미 익숙한 상태이므로 저항이 적다.

---

## 4. 전체 흐름 요약

```
[흡수기]  요구사항 거절 금지  →  JSON에 다 쌓기     →  태그 기록
    ↓
[관찰기]  겹치는 패턴 발견    →  설정 테이블 1개    →  온보딩 체크리스트
    ↓
[안정화]  타이밍 신호 확인    →  DB 정규화          →  기능 동결 + QA
```

---

## 5. 핵심 원칙

> **안정화는 "계획하는 것"이 아니라 "패턴이 보일 때 자연스럽게 하는 것"이다.**

지금 미리 완벽한 구조를 만들려고 시간을 쓰는 것보다,  
실제 업체 요구를 빠르게 흡수하면서 데이터를 모으는 게 훨씬 효율적이다.

---

## 부록. 현재 차량일지 작업내역 추가 요청 사례

이 문서의 배경이 된 구체적 변경 요청:

- 요소수 항목 삭제 요청
- 공공봉투 항목 삭제 요청
- 공공봉투 삭제 위치에 작업내역 3개 테이블 추가
  - 중량제봉투 및 음식물용기, 재활·자원 (kg 단위, 반입장소 선택)
  - 중량계 및 봉투 수거량 기계 (L 단위)
  - 대형폐기물 (점 단위)
- 반입장소는 관리자(CONTRACTOR_ADMIN)가 사전 설정 후 작업자가 선택
