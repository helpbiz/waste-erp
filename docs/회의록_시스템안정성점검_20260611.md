> **경영진 요약**
>
> 현재 서비스는 핵심 비즈니스 기능(출퇴근, 급여, TBM, 차량일지)이 동작하나,
> 역할 체크 누락으로 **WORKER 계정이 급여 승인·운행일지 자동생성을 실행할 수 있는**
> 권한 상승 취약점이 운영 중에 존재합니다.
> PII 암호화 폴백 버그는 AWS/Vault KMS 장애 1회만으로 전 직원의 주소·계좌번호가
> null로 소실되는 데이터 복구 불가 시나리오를 만들며,
> 근태 월마감·급여 확정 처리가 트랜잭션 없이 실행되어 부분 마감 상태가 고착될 수 있습니다.
> **총 43건의 Critical 이슈 중 P0 항목 7건을 먼저 해결한 후 단계적으로 안정화**하지 않으면
> 실제 운영 사고로 이어질 가능성이 높습니다.

#### P0 상세

**P0-1: 급여 승인 역할 체크 누락**
- **문제**: `POST /api/payroll/payslips/approve`에 role 검증이 없어, `PayrollPolicy.payslipApproverId`에 WORKER userId가 설정된 경우 해당 근로자가 전 직원 급여명세를 일괄 승인할 수 있음
- **위험**: 급여 승인은 법적 효력이 있는 노무 행위. 노동분쟁 시 증빙 무효화 가능
- **수정 방향**: 핸들러 최상단에서 `session.role`이 `CONTRACTOR_ADMIN` 또는 `SUPER_ADMIN`인지 확인 후 그 외 역할은 403 반환

**P0-2: 운행일지 자동생성 역할 체크 누락**
- **문제**: `POST /api/vehicle-logs/auto-generate`에 role 검증 없어, 인증된 WORKER가 동일 업체 내 임의 차량에 SUBMITTED 상태 운행일지를 생성하거나 `driverId`를 임의 지정 가능
- **위험**: 차량 운행 기록 위조, 기존 SUBMITTED 레코드 덮어쓰기, 차량 관련 법정 기록 훼손
- **수정 방향**: `CONTRACTOR_ADMIN` 이상 역할만 허용하는 role guard 추가

**P0-3: KMS 폴백 버그**
- **문제**: `lib/kms.ts`의 `getMasterKey()`가 AWS/Vault 장애 시 경고 로그 1줄만 남기고 `LocalKmsProvider`(JWT_SECRET 기반)로 폴백하여 잘못된 키를 캐시에 저장. 기존 암호화된 PII를 다른 키로 복호화 시 `catch` 블록에서 `null` 반환 → 주소·계좌번호·건강기록 조용히 소실
- **위험**: KMS 일시 장애 1회로 전사 PII 영구 소실처럼 보이는 운영 사고. 개인정보보호법 위반 및 급여이체 오류
- **수정 방향**: `getMasterKey()`에서 AWS/Vault 실패 시 폴백하지 않고 즉시 예외를 throw. `decryptField()`의 catch 블록도 null 반환 대신 오류를 상위로 전파

**P0-4: 근태 월마감·급여 처리 트랜잭션 누락**
- **문제**: `POST /api/attendance/finalize-month`의 근로자 루프와 `POST /api/payroll/payslips/import·prefill`의 다중 upsert가 `$transaction` 없이 실행됨
- **위험**: 월 급여 산정 기준이 되는 근태 마감이 불완전 상태로 고착. 재시도 시 `already_finalized` 409로 일부 근로자 재처리 자체 차단
- **수정 방향**: finalize-month, payslips/import, payslips/prefill의 멀티워커 루프 전체를 `prisma.$transaction()`으로 감싸기

**P0-5: 관리자 TBM 사진 업로드 버그**
- **문제**: `app/(admin)/safety/_safety-client.tsx`에서 `URL.revokeObjectURL()`이 `img.src` 할당 이전에 호출됨. Object URL 해제 후 이미지 로드 실패
- **위험**: TBM 현장 안전 기록 사진을 관리자가 확인할 수 없어 안전관리 감독 기능이 실질적으로 작동하지 않음
- **수정 방향**: `img.onload` 또는 `img.onerror` 콜백 내부로 `URL.revokeObjectURL()` 호출 이동

**P0-6: 하드코딩 초기 비밀번호**
- **문제**: `app/api/import/workers/route.ts`에 `'Qwer1234!'`가 초기 비밀번호로 하드코딩. 강제 변경 플래그(`mustChangePassword`)나 초기 로그인 리다이렉트 메커니즘 없음
- **위험**: 저장소 접근 권한자 전원이 이 비밀번호를 앎. 대량 계정에 즉시 로그인 가능
- **수정 방향**: 초기 비밀번호를 `crypto.randomBytes(12)`로 계정마다 개별 생성하거나, `mustChangePassword` 플래그 추가 및 최초 로그인 시 변경 강제

**P0-7: Puppeteer browserPromise 크래시 복구 불가**
- **문제**: 3개 PDF 엔드포인트가 공유하는 단일 `browserPromise` 싱글턴에 크래시 감지 로직 없음. 동시 PDF 요청 2~3개로 Chromium OOM 크래시 발생 시 `browserPromise`가 rejected 상태로 고착 → 컨테이너 재시작 없이 PDF 기능 영구 중단
- **수정 방향**: `getBrowser()` 내부에서 `browser.on('disconnected', () => { browserPromise = null; })` 핸들러 등록. `page.pdf()`에 30초 타임아웃 추가. p-limit으로 동시 2건 이하 제한

## 반복 버그 패턴 분석 — 구조적 원인

### 패턴 1: 방어적 파싱 없는 형변환 (18곳 이상)
- `BigInt()`, `Number()` 형변환이 유효성 검증 없이 18개 이상의 라우트에 산재
- `lib/ids.ts`에 `parseId()`가 이미 올바르게 구현되어 있음에도 미사용
- **근본 원인**: 팀 내 유틸 함수 사용 표준 부재
- **구조적 해결책**: ESLint 커스텀 룰 `no-direct-bigint` 도입 → 빌드 단계 차단

### 패턴 2: res.ok 체크 없는 fetch (8곳 이상)
- 관리자 클라이언트 파일 전반에 `res.ok` 확인 없이 `.json()` 직접 호출
- worker 측은 올바르게 구현되어 있어 코드 리뷰 범위 불균형
- **근본 원인**: PR 리뷰 체크리스트 없음

### 패턴 3: 사진 업로드 압축 미적용 (별도 발생)
- `_safety-worker-client.tsx` 수정 후 `_worker-client.tsx`에 동일 패턴 잔존
- 이번 세션에서 실제 사용자 오류 발생하여 발견·수정됨
- **근본 원인**: 유사 파일 간 수정 전파 누락 — 변경 시 grep 기반 동일 패턴 전파 확인 필요
