## 1. 현재 구조의 Trade-off

### 1.1 햄버거 메뉴의 알려진 Hidden Cost

UX 연구에서 반복적으로 확인된 문제점 3가지:

**발견율(Discoverability) 저하**
Nielsen Norman Group(2016, "Hamburger Menus and Hidden Navigation Hurt UX")은 햄버거 아이콘이 탭바 대비 사용자 탐색 속도를 평균 1.2배 느리게 만들고, 메뉴 항목의 발견율을 20-30% 낮춘다는 실험 결과를 발표했다. 보이지 않으면 존재를 인식하지 못한다.

**추가 클릭 비용**
햄버거 → 열림 → 항목 탭 → 닫힘 = 최소 2 인터랙션. 탭바 = 1 인터랙션. 하루 수십 번 전환 시 누적 마찰이 크다. 야외·이동 중에는 이 차이가 더 벌어진다.

**인지 부하**
드로어를 열었을 때 "어디에 뭐가 있는지" 기억해야 한다. 탭바는 항상 현재 위치를 시각적으로 보여준다(active indicator). 숨겨진 메뉴는 사용자가 앱 구조 전체를 머릿속에 지도로 유지해야 한다.

### 1.2 배민커넥트·카카오T·시프티의 공통 선택

docs/00-pm/mobile-ux-research.md §1.2 비교표 요약:

| 앱 | 네비게이션 | 햄버거 사용 |
|----|-----------|-----------|
| 배민커넥트 | 하단 탭바 4개 | 없음 |
| 카카오T | 하단 탭바 4개 | 없음 |
| 시프티 | 하단 탭바 4개 | 없음 |
| 당근알바 | 하단 탭바 5개 | 없음 |
| 쿠팡플렉스 | 상단 햄버거 | **있음** → 반면교사 사례로 명시됨 |

쿠팡플렉스만 햄버거를 쓰고, 리서치 문서에서 "반면교사 사례"로 분류되어 있다. 업계 수렴 방향은 명확하다.

**채택 이유 (배민커넥트 관점)**: 라이더는 이동 중 한 손으로 앱을 조작한다. 엄지가 자연스럽게 닿는 하단 탭바가 상단 햄버거보다 오조작 없이 빠르다. 출근 시작 → 배달 완료 → 수익 확인의 반복 패턴을 탭바 3개 탭으로 해결한다.

### 1.3 야외 환경미화원(40-60대) 추가 변수

현 사용자 컨텍스트에서 햄버거가 더 불리한 이유 3가지:

1. **장갑 착용**: 상단 좌측 햄버거 아이콘(~24x24px)은 작은 터치 타겟. 하단 탭은 flex-1로 분할되어 실질 너비 ~78px(390px ÷ 5).
2. **직사광선**: 화면 밝기가 낮을 때 작은 아이콘 구별이 어렵다. 탭바는 항상 노출되어 있어 위치 기억만으로도 조작 가능.
3. **디지털 리터러시**: 40-60대는 햄버거 아이콘을 직관적으로 인식하지 못할 수 있다. 시프티 사용자 리뷰에서 "나이 많은 직원들도 불편함 없이 적용"이 탭바 구조의 직접 근거로 언급된다.

## 2. 대안 패턴 5가지 비교

### Option A — Bottom Tab 6-7개 (사용자 제안: 모두 표시)

```
┌─────────────────────────────┐
│  [AppBar: 이름 + 역할]       │
├─────────────────────────────┤
│                             │
│  [본문 콘텐츠]              │
│                             │
├─────────────────────────────┤
│ 홈 출퇴근 민원 안전 실적 휴가│  ← 6개
└─────────────────────────────┘
```

RAPID 포함 시:
```
│홈 출퇴근 민원 안전 실적 휴가 경로│  ← 7개
```

**장점**
- 햄버거/드로어 완전 제거. 1탭 직접 접근.
- AppBar 좌측 햄버거 버튼도 제거 → 레이아웃 단순화.

**단점**
- 6개 분할 시 탭 너비 65px, 7개 시 55px. 라벨 글자가 8-9px로 줄어 야외 가독성 위반 (최소 12px 기준).
- "추천경로"는 RAPID 직책만 노출해야 하므로 역할별 탭 수가 달라진다 → 레이아웃 불일치감.
- iOS/Android HIG 모두 탭 5개 이하를 권고. 6-7개는 비표준 패턴.

**평가**: 의도는 옳지만 7개는 과부하. 라벨 가시성 문제로 야외 환경 부적합.

---

### Option B — 2-row Bottom Tab (4×2 격자)

```
┌─────────────────────────────┐
│  [AppBar]                   │
├─────────────────────────────┤
│  [본문 콘텐츠]              │
├─────────────────────────────┤
│ [홈] [출퇴근] [민원] [안전] │  ← 1행
│ [실적] [휴가] [프로필] [경로]│  ← 2행 (RAPID: 경로 포함)
└─────────────────────────────┘
```

**장점**
- 모든 메뉴 상시 노출.
- 드로어·햄버거 완전 제거.

**단점**
- 탭바 높이 128px. 본문 콘텐츠 영역 심각 축소 (iPhone 14 Pro: 700px 중 ~180px이 탭바+appbar).
- 2행 탭바는 어떤 주요 앱도 채택하지 않는 비표준 패턴. 학습 비용 높음.
- RAPID/비RAPID 간 행 구성이 달라져 레이아웃 점프 발생.

**평가**: 기술적으로 가능하지만 실사용 데이터가 없는 실험적 패턴. 권장 불가.

### Option C — Bottom Tab 5 + 우측 상단 아바타 (드로어 대신)

```
┌─────────────────────────────┐
│  [AppBar: 이름]   [프로필●] │  ← 우측 아바타 버튼 → 프로필 직접 이동
├─────────────────────────────┤
│  [본문 콘텐츠]              │
├─────────────────────────────┤
│  홈 | 출퇴근 | 민원 | 안전 | 더보기 │
│                   ↓ BottomSheet
│              실적 / 휴가 / 경로(RAPID)
└─────────────────────────────┘
```

**장점**
- 프로필을 상단 아바타로 분리 → 드로어 내부 항목이 3-4개로 감소.
- Instagram·GitHub 등 주요 앱의 확립된 패턴.
- 드로어 내 항목이 줄어 "더보기" 탭의 부담 감소.

**단점**
- 상단 아바타는 엄지 도달 범위 밖 (상단 우측은 가장 멀다).
- 드로어를 완전 제거하지는 못함 — 실적/휴가/경로가 여전히 숨겨진다.
- 햄버거 버튼 위치 문제만 약간 개선, 근본 해결 아님.

**평가**: 현재보다 낫지만 핵심 문제(숨겨진 항목) 미해결.

### Option D — Bottom Tab 4 + 홈 페이지 그리드 메뉴 (Full set on Home)

```
┌─────────────────────────────┐
│  [AppBar: 이름·역할]         │
├─────────────────────────────┤
│  ┌─────────┐ ┌─────────┐   │
│  │ 출퇴근  │ │  민원   │   │  ← 홈 그리드 (자주쓰는 메뉴)
│  │  GPS    │ │  사진   │   │
│  └─────────┘ └─────────┘   │
│  ┌─────────┐ ┌─────────┐   │
│  │  안전   │ │  실적   │   │
│  └─────────┘ └─────────┘   │
│  ┌─────────┐ ┌─────────┐   │
│  │  휴가   │ │ 프로필  │   │
│  └─────────┘ └─────────┘   │
│  [경로] (RAPID only)        │
├─────────────────────────────┤
│   홈  |  출퇴근  |  안전  |  나  │  ← 핵심 4개 고정
└─────────────────────────────┘
```

**장점**
- 탭바 4개 → 탭 너비 여유 (97px). 라벨 크기 충분.
- 홈에서 전체 메뉴 1탭 접근 — 그리드 패턴은 카카오T·당근알바 차용.
- 드로어·햄버거 완전 제거.
- RAPID 경로 탭은 홈 그리드에만 조건부 노출 → 레이아웃 안정.

**단점**
- 홈이 "런처(launcher)" 역할 겸용 → 홈의 대시보드 기능과 역할 혼재.
- 실적/휴가/프로필에서 돌아오려면 항상 탭바 '홈' 탭 거쳐야 함 → 탐색 깊이 증가.
- 현재 홈 페이지 구조 변경 필요.

**평가**: 메뉴 수가 적은 앱(6개 이하)에 적합. 8개이고 실적/휴가 사용 빈도가 있는 경우 홈 그리드만으로는 피로도 가능.

### Option E — 현재 (5 + 햄버거 Drawer) — 현상 유지

```
┌─────────────────────────────┐
│  ☰ [이름·역할]              │  ← 좌측 햄버거
├─────────────────────────────┤
│  [본문 콘텐츠]              │
├─────────────────────────────┤
│ 홈 | 출퇴근 | 민원 | 안전 | 더보기 │
│                   ↓ Drawer
│     실적 / 휴가 / 프로필 / 경로(RAPID) / 로그아웃
└─────────────────────────────┘
```

**장점**
- 이미 구현 완료 — 변경 비용 없음.
- 탭바 5개는 당근알바와 동일 — 한국 시장 익숙.
- `isRapid` 서버 로직 재사용 가능.

**단점**
- AppBar에 햄버거 + 탭바에 "더보기" — 두 가지 진입점이 모두 드로어를 열어 중복 혼란.
- 실적/휴가/프로필이 숨겨져 있어 발견율 낮음.
- 40-60대 사용자에게 "더보기"의 의미가 불명확.

4. RAPID 경로 탭은 홈 그리드 내 `isRapid` 조건부 노출 — 탭바 레이아웃 불변.
5. 탭 4개 → 너비 97px, 아이콘 24px + 라벨 12px = 여유 충분.

## 5. 사용자 의견 검증

### "굳이 햄버거?" — 맞다 (Yes)

**결론: 현재 구조에서 햄버거 메뉴는 정당화되기 어렵다.**

근거:

1. **8개 메뉴 중 4개가 숨겨짐**: 실적·휴가·프로필·경로가 드로어 안에 있다. 발견율 연구 기준으로 이 4개는 사용자에게 "없는 것처럼" 느껴진다.

2. **진입점 중복**: 햄버거(AppBar 좌측) + "더보기 탭"(탭바 우측) — 둘 다 같은 드로어를 연다. 사용자는 "왜 두 군데서 열리는 같은 것이 있나?" 혼란.

3. **드로어 필요 조건 미충족**: 드로어(Drawer)는 ① 메뉴가 10개 이상이거나 ② 계층적 하위 메뉴가 있거나 ③ 화면 공간 제약이 극심할 때 사용한다. 워커 앱은 8개 단층 메뉴 — 어느 조건도 해당하지 않는다.

4. **모바일 트렌드 역행**: 한국 주요 필드워커 앱 5개 중 4개가 하단 탭바만 사용. 햄버거는 쿠팡플렉스만 사용하고 리서치에서 "반면교사"로 분류.

### 워커 사용 빈도 추정

| 메뉴 | 예상 사용 빈도 | 배치 권고 |
|------|------------|---------|
| 홈 | 매일 수회 (앱 진입 기본) | 탭바 |
| 출퇴근 | 매일 2회 (출근·퇴근) | 탭바 |
| 민원 | 수시 (현장 촬영) | 탭바 |
| 안전 | 매일 1회 (TBM) | 탭바 |
| 실적 | 주 1-3회 (실적 확인) | 홈 그리드 |
| 휴가 | 월 수회 (신청·조회) | 홈 그리드 |
| 프로필 | 월 1-2회 | 홈 그리드 or 아바타 |
| 추천경로 | 매일 1회 (RAPID만) | 홈 그리드 |

출퇴근·민원·안전이 워커의 핵심 반복 업무 — 탭바 고정이 타당. 실적·휴가는 참조형 접근으로 홈 그리드 1탭 접근으로 충분. "더보기 + 드로어"의 2단계 접근이 불필요한 이유다.
