## 1. Value Proposition — JTBD 6-Part (Tony Ulwick 변형)

> JTBD 6-Part = **Who / When / Want / So that / Replace / Outcome**

### 1.1 핵심 페르소나 (worker — 50~60대 환경미화원)

| 차원 | 내용 |
|------|------|
| **Who (누가)** | 평균 50~60대, 시니어 환경미화원 — 일반 미화원 + 기동반(RAPID) 두 하위 타입 |
| **When (언제 — 트리거 상황)** | 매일 새벽 5시 출근 / 폭염·우천·야간 시야 제약 / 장갑 착용 / 한 손 조작 / 지자체 지급폰 또는 본인폰의 PWA로 |
| **Want (원하는 것 — 기능적 욕구)** | 출근 체크인, 안전점검, 민원 응답, (RAPID는) 최적 수거 경로를 **헤매지 않고 1~2탭으로** 끝내고 싶다 |
| **So that (왜 — 정서/사회적 욕구)** | 늦지 않고 / 사고 없이 / 시민 컴플레인 없이 / "젊은 사람만 쓰는 앱"이 아닌 "내가 쓸 수 있는 앱"이라는 자존감을 유지하면서 |
| **Replace (대안)** | 종이 점검표 + 카카오톡 단톡방 / 종전 ERP 데스크탑 화면 / 무자력으로 못 찾고 다른 직원에게 묻기 |
| **Outcome (성공 지표)** | 출근→안전점검→첫 작업 시작까지 5분 이내 / 1주차 자력 사용 100% / "이 앱은 큰 글씨라 좋다"는 자발적 평가 |

### 1.2 보조 페르소나 — Company / Manager / Muni 압축

| Role | When | Want | So that |
|------|------|------|---------|
| **Company 사장** | 출퇴근길/현장 이동 중 짧은 짬 | 오늘 운영 한눈 + 결재 처리 | 사무실 가지 않아도 회사가 굴러간다 |
| **Manager** | 사무실 + 현장 왕복 | 일일 점검 + 알림 즉시 인지 | 사고/민원이 묻혀 비상이 되지 않게 |
| **Muni 담당** | 본청 사무실 + 현장 점검 | 위탁사 KPI + 민원 흐름 모니터링 | 감사 / 시민 신뢰를 잃지 않게 |

### 1.3 한 줄 VP 문장

> **"고령자도 5분 만에 익힌다. 4-role 모두에게 같은 셸. PWA지만 앱처럼 잠긴 느낌. WCAG 2.1 AAA 기본."**

## 2. Lean Canvas (Ash Maurya)

본 이니셔티브는 신규 제품 출시가 아닌 **운영 제품의 UX 마스터링**이지만, 가치 흐름을 명확히 하기 위해 Lean Canvas로 정리한다.

| # | 블록 | 내용 |
|---|------|------|
| 1 | **Problem (문제)** | (a) 고령 미화원이 핵심 액션을 자력으로 못 한다 (가독성·터치). (b) PWA지만 "앱답지 않다"는 신뢰도 결손. (c) 4-role × 페이지마다 셸이 달라 인지부하. (d) 부분 패치 누적 — 부분만 고치면 전체가 또 깨진다. |
| 2 | **Customer Segments** | **Early Adopters**: 환경미화원 50~60대 (북극성). **Beachhead 후보**: 1개 위탁사 × 1개 지자체 권역(20~50명 미화원). **확장**: 전국 250+ 지자체 × 위탁사. |
| 3 | **Unique Value Proposition** | "고령자도 5분, 4-role 한 셸, PWA지만 잠긴 앱, WCAG AAA 기본." |
| 4 | **Solution** | (a) WCAG 2.1 AAA 디자인 토큰. (b) 4-role 통합 `<AppShell>`. (c) viewport 잠금 + safe-area. (d) 단일 로그인 + role 자동 라우팅. (e) RBAC 메뉴 (RAPID 분기 포함). (f) 로그아웃 동선 재설계. |
| 5 | **Channels** | 지자체 발주 사업 → 위탁사 직접 도입. 사용자 측면: PWA "홈화면 추가" 가이드 + 현장 5분 교육 영상 + 사용자 설명서 (`사용자_설명서.md`). |
| 6 | **Revenue Streams** | (운영 ERP의 일부이므로) 직접 매출은 없음. **간접 매출**: ERP 갱신 계약 유지·확장, 신규 위탁사 도입 시 진입장벽 제거(Sales Enabler). |
| 7 | **Cost Structure** | 디자인 시스템 도입(에이전트/디자이너 시간), 통합 셸 리팩터링, QA(시니어 사용성 테스트, axe E2E), 시각적 회귀 테스트. |
| 8 | **Key Metrics** | (a) 시니어 SUS ≥ 80, (b) DAU/MAU ≥ 0.7 (worker), (c) 첫 핵심 액션 성공률 ≥ 95%, (d) WCAG AAA Lighthouse ≥ 95, (e) "앱 같다" 정성 평가, (f) 운영 단톡방의 "어떻게 해요?" 문의 50% 감소. |
| 9 | **Unfair Advantage** | (a) 운영 중 실데이터/실사용자 — 가설 즉시 검증 가능. (b) 4-role 다 가진 ERP는 흔치 않음 — 통합 셸 자체가 차별. (c) 한국 시니어 미화원에 특화된 "큰 글씨/고대비" 정책을 제품 헌법으로 — 후발 SaaS가 따라오기 어려움. |

## 3. SWOT 분석 (이니셔티브 관점)

|  | 도움 (Helpful) | 방해 (Harmful) |
|--|----------------|----------------|
| **내부 (Internal)** | **S — Strengths**<br>• 운영 중 PWA — 실사용자/실데이터 즉시 가용<br>• Next.js App Router + Tailwind + Service Worker 갖춤<br>• Wave 1~4에서 부분 결함을 이미 식별·기록<br>• Playwright 다해상도 visual regression + axe a11y 셋업 완료 | **W — Weaknesses**<br>• 부분 패치 누적으로 셸/패턴 비일관<br>• 로그아웃·viewport 같은 기본 결함이 운영 중 존재<br>• 디자인 시스템(토큰 체계) 부재<br>• role × 페이지 매트릭스 검증 자동화 부족 |
| **외부 (External)** | **O — Opportunities**<br>• 한국 환경미화원 인구 통계상 "시니어 친화"가 카테고리 룰이 됨<br>• PWA 표준(viewport/safe-area, manifest) 성숙<br>• WCAG 2.1 AAA 도구(axe, Lighthouse, Stark)가 완비<br>• 공공 발주에서 접근성 의무 강화 추세 | **T — Threats**<br>• 부분 패치 유혹(시간 압박)으로 다시 표면만 고칠 위험<br>• role 별 요구가 충돌 시 셸 통일이 깨질 위험<br>• 시니어 사용자 일부는 "큰 글씨" 자체를 거부할 가능성(존중 문제) → UX 리서치 필요<br>• 지자체별 브랜드 색상 커스터마이즈 요구 |

### 3.1 전략 매트릭스 (TOWS)

| 전략 | 내용 |
|------|------|
| **SO (강점×기회)** | 실사용자에게 매주 사용성 테스트 → AAA 기준을 "공공 발주 차별화 자산"으로 마케팅 자료화 |
| **WO (약점×기회)** | 부재한 디자인 시스템을 "한국 시니어 친화 운영 ERP 토큰 체계"로 처음 만든다 — 카테고리 정의 시도 |
| **ST (강점×위협)** | 시각적 회귀 + axe E2E를 CI에 강제하여 부분 패치 회귀를 시스템적으로 차단 |
| **WT (약점×위협)** | "처음부터 다시 그린다" 사고를 정책화 — 본 이니셔티브 동안은 부분 패치 PR 거부 |

## 4. Pricing / BMC 핵심 (간소화)

CleanERP는 운영 SaaS의 일부이므로 본 이니셔티브가 직접 가격에 영향을 주진 않으나, **가치 가격 결정 요인**은 다음과 같이 갱신된다.

| 가치 차원 | 본 이니셔티브의 기여 |
|-----------|---------------------|
| **인력 절감** | 단톡방 "어떻게 해요?" 문의 감소 → 관리자 응대 시간 감소 |
| **사고 방지** | 안전점검을 자력으로 끝내는 비율 상승 → 산업재해 / 시민 민원 감소 |
| **갱신 락인 (Retention)** | "큰 글씨/AAA"를 다른 ERP로 옮기기 어렵게 만드는 차별점 — 갱신 협상력↑ |
| **공공 발주 점수** | 접근성 등급 AAA 인증/Lighthouse 결과 → RFP 가점 |
| **신규 위탁사 도입 속도** | 5분 자력 학습 → 도입 마찰 비용↓ → Sales 사이클 단축 |

### 4.1 BMC 핵심 9블록 압축 (Lean Canvas와 중복은 생략)

- **Key Activities**: 디자인 토큰 정의, 통합 셸 구축, 사용성 테스트, axe/visual CI 강화
- **Key Resources**: 시니어 사용자 패널 (5~10명), 위탁사 1곳 베타 권한, Tailwind/Next.js 셋업
- **Key Partners**: 베타 위탁사, 지자체 1곳(피드백 채널), 접근성 컨설턴트(선택)
- **Customer Relationships**: 현장 5분 교육 + 인앱 onboarding 카드 + Slack/카카오톡 베타 채널

## 5. 추가 프레임워크 (맥락 적합)

### 5.1 Kano Model — 시니어 미화원 관점

| 카테고리 | 본 이니셔티브 기능 |
|----------|------------------|
| **Must-Have (당연)** | 글씨 크기, 터치타겟, 핀치 줌 잠금, 로그아웃 시인성, 단일 로그인 |
| **Performance (선형)** | 대비 7:1, 셸 일관성, 오프라인 회복력 |
| **Delight (감동)** | 야외 가독성 모드, 햇빛/야간 자동 전환, "잠긴 PWA" 신뢰감, 한 손 풀-와이드 CTA |
| **Reverse (역)** | 화려한 애니메이션, 영문 약어, 아이콘만 있는 메뉴 (라벨 없는 아이콘은 시니어에게 -점수) |

### 5.2 Hooked Model — 매일 자발적 사용 만들기 (Nir Eyal)

| 단계 | 본 이니셔티브 적용 |
|------|------------------|
| **Trigger (외부)** | 출근 시간 알림 (PWA push), 카톡 단톡방 → "앱에서 확인" 링크 |
| **Trigger (내부)** | "오늘 안전점검 했지?" 불안감 → 앱 열기 |
| **Action** | 1탭 출근 체크인, 1탭 안전점검 시작 — Fogg's behavior 모델의 "능력 = 쉽게" 만족 |
| **Variable Reward** | 오늘 처리량/내 순위 카드, 가끔 칭찬 메시지 ("○○님 30일 무사고") |
| **Investment** | 점검 기록 누적, 본인 프로필 사진/이름, 즐겨찾기 경로(RAPID) — 떠나기 어려운 자산 |

### 5.3 RICE 우선순위 (이니셔티브 내부 솔루션 정렬)

| 솔루션 | Reach | Impact | Confidence | Effort | RICE |
|--------|-------|--------|-----------|--------|------|
| WCAG AAA 디자인 토큰 | 4-role 전원 (1.0) | 3 (대) | 0.9 | 5주 | **2.16** |
| 4-role 통합 셸 | 4-role 전원 (1.0) | 3 (대) | 0.85 | 6주 | **1.70** |
| viewport 잠금 + safe-area | 전원 (1.0) | 2 | 0.95 | 0.5주 | **7.60** |
| 단일 로그인 + role 라우팅 | 전원 (1.0) | 2 | 0.9 | 1주 | **3.60** |
| 로그아웃 시인성 재설계 | 전원 (1.0) | 3 | 0.95 | 0.5주 | **11.40** |
| RBAC 메뉴 (RAPID 분기) | worker (0.5) | 2 | 0.9 | 1주 | **1.80** |

> **즉시 실행 후보 (RICE 큰 순)**: 로그아웃 시인성(11.4) → viewport 잠금(7.6) → 단일 로그인(3.6) → AAA 토큰(2.16) → RBAC(1.8) → 통합 셸(1.7).
> **단, 이니셔티브 정책상 "처음부터 다시" — 토큰 + 통합 셸을 먼저 깔고 그 위에서 나머지를 구현하는 순서를 권고** (PRD §실행계획).

## 6. 의도된 트레이드오프 (명시)

| 결정 | 포기한 것 | 얻는 것 |
|------|---------|---------|
| WCAG 2.1 **AAA** (AA 아님) | 일부 미려한 색상 조합 (대비 미달분), 폰트 크기 자유도 | 시니어/저시력 사용자 자력 사용 + 공공 발주 차별 |
| viewport `maximum-scale=1` + `user-scalable=yes` | 자유로운 핀치 줌 + 검색 엔진 SEO 일부 (PWA에는 무관) | "앱처럼 잠긴" 신뢰도 + 줌 접근성 호환 |
| 단일 통합 셸 | role별 화려한 차별화, 페이지별 자유로운 레이아웃 | 인지부하 감소, 유지보수 비용 1/4 |
| 단일 로그인 + 자동 라우팅 | role 진입의 "초대받은 느낌" | 신규 사용자 학습 비용↓, 로그인 화면 1개로 압축 |
| 부분 패치 거부 정책 | 단기 빠른 하나 고치기 | 누적 회귀 차단, 본 이니셔티브 종료 후 안정성 |
