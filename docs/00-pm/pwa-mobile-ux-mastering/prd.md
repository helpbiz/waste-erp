## Executive Summary

| 관점 | 요약 |
|------|------|
| **문제 (Problem)** | 50~60대 환경미화원과 4-role 사용자가 CleanERP PWA에서 (a) 안 보이고(가독성/대비/터치), (b) 잠긴 느낌이 없고(자유 줌), (c) 페이지마다 셸이 달라(인지부하) 자력 사용을 못한다. Wave 1~4 부분 패치는 누적 회귀를 만들었다. |
| **솔루션 (Solution)** | 처음부터 다시: WCAG 2.1 AAA 디자인 토큰 + 4-role 통합 `<AppShell>` + viewport 잠금(maximum-scale=1, user-scalable=yes) + 단일 로그인 + role 자동 라우팅 + RBAC 메뉴(RAPID 분기) + 로그아웃 동선 재설계. 부분 패치 거부 정책. |
| **기능 UX 효과 (Function UX Effect)** | 시니어 SUS ≥ 80, 자력 사용 100% 5분 학습, "앱 같다" 정성 평가 ≥ 80%, 단톡방 문의 50%↓, Lighthouse 접근성 ≥ 95, axe E2E 위반 0건. |
| **핵심 가치 (Core Value)** | "고령자도 5분, 4-role 한 셸, PWA지만 잠긴 앱, WCAG AAA 기본." 한국 시니어 친화 운영 ERP의 카테고리 정의 시도 + 공공 발주 RFP 차별 자산 + 신규 위탁사 도입 마찰 제거. |

## 1. ICP (Ideal Customer Profile)

### 1.1 ICP 정의

> **"한국 광역시·중규모 시 단위 지자체와 위탁계약 중인 위탁대행사 중, 환경미화원 20~50명을 운영하고 평균 연령 50대 이상이며, 지자체에서 디지털 운영 전환을 추진하는 곳."**

| ICP 차원 | 적합 조건 |
|----------|-----------|
| 위탁사 규모 | 환경미화원 20~50명 (스몰~미드) |
| 미화원 인구 | 평균 50대 이상, 시니어 비율 60%+ |
| 지자체 성향 | 디지털 운영 전환 + 접근성 정책 강화 추세 |
| ERP 도입 의지 | 종이/카톡 → 시스템 전환 의지가 있는 사장(P3 박철수 페르소나) |
| 디바이스 환경 | 지자체 지급폰 또는 본인폰 — Android 우세 + 일부 iOS |

### 1.2 페르소나 매핑

- **북극성**: P1 김순기 (worker 일반 미화원, 58세)
- **확장 조건**: P2 이순임 (RAPID), P3 박철수 (company), P4 정혜영 (muni) 모두 같은 셸로 만족시킬 수 있어야 ICP 적합

## 2. Beachhead Segment (Geoffrey Moore 4-criteria 채점)

### 2.1 후보 세그먼트 평가

| Criteria | 후보 1: **현재 운영 중 위탁사 1곳 (베타)** | 후보 2: 신규 도입 위탁사 5곳 | 후보 3: 지자체 1곳 (위탁사 다수) |
|----------|--------------------------------------------|------------------------------|----------------------------------|
| **1. 식별 가능 (Targetable)** | ✅ 매우 (운영 중) | △ 영업 필요 | ○ 가능 |
| **2. 도달 가능 (Reachable)** | ✅ 직접 채널 보유 | △ 콜드 콜/RFP | ○ 지자체 미팅 |
| **3. 명확한 동기 (Compelling Reason to Buy)** | ✅ 사용자 보고 결함 직접 해결 | ○ "AAA 마케팅" 어필 | ○ "감사·민원 통제" |
| **4. 완전한 솔루션 가능 (Whole Product)** | ✅ ERP 본체 + 디자인 토큰 + 셸 + 매뉴얼 + 5분 교육 가능 | △ 신규 도입 + 데이터 이관 부담 | △ 여러 위탁사 협조 필요 |
| **합계** | **4/4** | 1.5/4 | 2/4 |

### 2.2 Beachhead 결정

> **베타 위탁사 1곳 (운영 중) — 환경미화원 30명 ± / 일반:RAPID = 9:1 / 평균 연령 50대 후반 / Android 70% iOS 30% / 본 이니셔티브 결과 1차 검증 대상**

**근거**: (a) 사용자 보고 결함을 직접 해결하므로 가치 입증이 즉시 가능, (b) 직접 채널, (c) 본 이니셔티브 종료 후 다른 위탁사로 확산할 레퍼런스 케이스가 된다.

### 2.3 Beachhead 성공 후 확산 경로

```
[Beachhead] 베타 위탁사 1곳 (30명)
   │
   ▼ 6개월 검증 후 레퍼런스화
[1단계 확산] 같은 지자체 내 다른 위탁사 2~3곳 (60~150명)
   │
   ▼ 12개월 후 케이스 스터디 자료
[2단계 확산] 인접 지자체 5~10곳 (500~1,000명)
   │
   ▼ 18~24개월 RFP 제안서에 AAA/시니어 친화 자산 활용
[3단계 확산] 광역 단위 진입 (SOM 하단 도달)
```

## 5. Pre-mortem (선제적 실패 시나리오)

> **방법**: 6개월 후 본 이니셔티브가 실패했다고 가정하고, 그 원인을 미리 적는다 (Gary Klein).

### 5.1 Top-3 위험

#### 위험 1 (Critical): "처음부터 다시" 정책이 깨져 부분 패치가 다시 시작된다

- **시나리오**: 운영 중 핫이슈(특정 페이지 결함)가 발생, 시간 압박으로 셸/토큰 구축 전에 패치 PR이 머지됨. Wave 5/6 회귀 누적.
- **확률**: 높음
- **영향**: 본 이니셔티브 자체 무효화
- **사전 완화**: (a) "PR 본문 체크리스트"에 "이 PR은 디자인 토큰을 통해 구현되었는가?" 항목 강제. (b) 운영 핫이슈는 별도 hotfix 브랜치 + 토큰 도입 후 즉시 흡수 약속. (c) PM/CTO 주간 점검에서 부분 패치 발견 시 즉시 회수.

#### 위험 2 (High): viewport 잠금이 지도(leaflet)/이미지 컴포넌트 줌과 충돌

- **시나리오**: RAPID '최적경로계산'의 leaflet 지도, 보고서 이미지, 첨부 파일 미리보기에서 "확대 안 됨" 컴플레인.
- **확률**: 중-높
- **영향**: RAPID 페르소나 핵심 가치 훼손
- **사전 완화**: (a) viewport 잠금은 페이지 단위가 아닌 **컴포넌트 내부 줌**과 명확히 분리. (b) leaflet은 자체 핀치 줌 활성. (c) 이미지/PDF 미리보기는 전용 모달 내부에서 컴포넌트 줌 제공. (d) 디자인 시스템 토큰에 "Zoomable Region" 컴포넌트 추가.

#### 위험 3 (High): 시니어 사용자 일부가 "큰 글씨"를 거부 (존중 문제)

- **시나리오**: "노인 취급한다"는 정서적 거부감 — 일부 50대 워커가 "글씨 너무 크다, 우습다"는 평을 함.
- **확률**: 중
- **영향**: 채택률 저하 + 부정 입소문
- **사전 완화**: (a) 본문 18px는 기본, 사용자별 16~22px 조절 옵션. (b) "큰 글씨 모드"라 부르지 말고 "기본"이라 부른다. (c) 시각 디자인을 시니어 전용처럼 보이지 않게 — 토스/배민커넥트와 동급의 미려함 유지.

### 5.2 추가 위험 (모니터링)

| 위험 | 확률 | 영향 | 완화 |
|------|------|------|------|
| AAA 색상 제약이 지자체 브랜드와 충돌 | 중 | 중 | 디자인 토큰을 지자체별 override 가능하게 (단, 대비 검증 자동) |
| 4-role 통합 셸이 일부 페이지에 안 맞음 | 중 | 중 | 셸의 escape hatch (`<FullScreen>` 컴포넌트) 명시적으로 정의 |
| PWA 푸시 권한 거부율 높음 | 중 | 중 | onboarding 3장에서 "왜 알림이 필요한지" 큰 글씨로 설명 |
| 단일 로그인 화면이 role을 잘못 라우팅 | 낮 | 높 | 라우팅 후 첫 화면에 "○○ role로 로그인됨" 큰 배너 + 1탭 변경 옵션 |

## 6. User Stories (INVEST 검증)

### 6.1 Epic 1 — 디자인 토큰 / WCAG AAA

- **US-1.1**: 사용자(any)로서 본문 18px+ / 대비 7:1 / 터치타겟 44×44px+를 보장받고 싶다, 그래야 노안에도 자력으로 사용한다.
  - INVEST: I ✅ / N ✅ / V ✅ / E (M) / S ✅ / T (Lighthouse ≥ 95, axe 0)
- **US-1.2**: 디자이너/개발자로서 색·폰트·간격·라운드·그림자가 단일 토큰 정의로 관리되길, 그래야 부분 패치 회귀를 차단한다.

### 6.2 Epic 2 — 4-role 통합 AppShell

- **US-2.1**: 사용자(any)로서 어느 페이지에서든 같은 위치(좌상단=홈/뒤로, 우상단=프로필 아바타, 하단=role별 5탭)에 같은 의미가 있길, 그래야 매번 헤매지 않는다.
- **US-2.2**: worker로서 풀-와이드 1차 CTA가 화면 하단에 있길, 그래야 장갑 끼고 한 손으로 누른다.
- **US-2.3**: company/muni로서 모바일 홈에서 KPI 카드 3장으로 한눈에 운영을 보고 싶다.

### 6.3 Epic 3 — 단일 로그인 + role 자동 라우팅

- **US-3.1**: 신규 사용자로서 4-role 어느 사람이든 같은 로그인 화면에 진입해서 ID/PW로 로그인하면 내 role 홈으로 자동 라우팅되길, 그래야 진입 분기를 학습할 필요가 없다.
- **US-3.2**: 사용자로서 라우팅 직후 "○○ role로 로그인됨" 큰 배너를 보고, 다른 role로 잘못 들어왔으면 1탭으로 로그아웃하고 싶다.

### 6.4 Epic 4 — viewport 잠금 + PWA 셸

- **US-4.1**: 사용자(any)로서 핀치 줌으로 화면이 끌리지 않길(`maximum-scale=1`), 그러나 OS 줌 접근성은 살아 있길(`user-scalable=yes`).
- **US-4.2**: RAPID로서 leaflet 지도와 이미지 미리보기는 컴포넌트 내부에서 핀치 줌이 살아 있길.
- **US-4.3**: 사용자(any)로서 status bar 색상, 스플래시, 홈화면 추가 가이드가 잘 갖춰진 PWA 신뢰감을 받고 싶다.

### 6.5 Epic 5 — 로그아웃 동선 재설계

- **US-5.1**: 사용자(any)로서 우상단 프로필 아바타(44×44px+) → 1탭 시 시트로 펼쳐지는 메뉴에 "로그아웃" 라벨이 본문 18px / 대비 7:1로 보이길.
- **US-5.2**: 지자체 지급폰을 다음 직원에게 넘길 때, "로그아웃 후 폰 잠금"까지 1탭 + 1확인으로 끝내고 싶다.

### 6.6 Epic 6 — RBAC 메뉴 + RAPID 분기

- **US-6.1**: RAPID worker로서 '최적경로계산' 메뉴/탭이 첫 화면 1탭 위치에 있길, 그러나 일반 worker에겐 보이지 않길.
- **US-6.2**: 일반 worker가 RAPID 전용 URL을 직접 입력해도 진입이 차단되길 (서버측 가드).

### 6.7 INVEST 자가 점검 (요약)

모든 US는 **Independent / Negotiable / Valuable / Estimable / Small / Testable**을 충족한다. Epic 6의 US-6.2는 보안성과 직결되므로 Plan 단계에서 우선순위 P0.

### 8.2 시나리오 (모바일 viewport: 375 / 393 / 360 / 768)

#### TS-01: 시니어 자력 출근 체크인 (P1 페르소나)

- **Given** 신규 worker 사용자가 PWA 첫 진입 (홈화면 추가 안 한 상태)
- **When** ID/PW로 로그인 → worker 홈 진입 → "출근하기" 풀-와이드 버튼 1탭
- **Then** (a) 위치 자동 첨부, (b) 출근 시각 기록, (c) 큰 confirmation 카드 표시, (d) 1분 이내 완료
- **Assertions**: 폰트 18px+, 터치타겟 44px+, 대비 7:1 (axe), 풀-와이드 버튼 width 100% - safe-area
- **자동화**: `e2e/onboarding-checkin.spec.ts` (예시 경로)

#### TS-02: 로그아웃 시인성 (사용자 보고 결함 직접 검증)

- **Given** 어느 role 어느 페이지든
- **When** 우상단 프로필 아바타 (44×44px+) 1탭
- **Then** Sheet가 열리고 "로그아웃" 라벨이 본문 18px / 대비 7:1 / 터치 44px+로 명확
- **Assertions**: axe 위반 0, visual snapshot 일치
- **자동화**: `e2e/logout-visibility.spec.ts`

#### TS-03: viewport 잠금 + 줌 접근성 호환

- **Given** PWA 진입 (iOS Safari standalone, Android Chrome standalone)
- **When** 사용자가 핀치 줌 시도 / OS 글씨 크기를 최대로 설정
- **Then** (a) 핀치 줌은 동작 안 함 / (b) OS 글씨 크기는 반영됨 / (c) leaflet 지도는 컴포넌트 내부 핀치 줌 동작
- **Assertions**: meta viewport 검증 (`maximum-scale=1, user-scalable=yes`), leaflet 줌 컨트롤 동작

#### TS-04: 단일 로그인 → role 자동 라우팅

- **Given** 4-role 각 1명 (worker/RAPID/company/manager/muni)
- **When** 같은 로그인 화면에서 ID/PW 입력 → 로그인
- **Then** 각 role의 홈으로 자동 라우팅 + "○○ role로 로그인됨" 큰 배너 표시
- **Assertions**: 라우팅 정확도 100%, 배너 노출 100%, 1초 이내

#### TS-05: RAPID 분기 / 일반 worker 차단

- **Given** 일반 worker 로그인 상태
- **When** 메뉴 탐색 / `/worker/optimal-route` 직접 URL 진입 시도
- **Then** (a) 메뉴/탭에 '최적경로계산' 미노출, (b) 직접 URL 진입 시 서버측 차단 + 404 또는 안내 페이지
- **Assertions**: DOM에 해당 메뉴 0개, 서버 응답 200/404 (정책 결정), RAPID 사용자는 정상 진입

#### TS-06: 4-role 셸 일관성 (visual regression)

- **Given** 4-role × 핵심 페이지 N개
- **When** 각 페이지를 375/393/360/768 viewport에서 렌더
- **Then** 헤더(좌상단 동작/우상단 아바타), 하단 5탭, FAB, 토큰 색상이 일치
- **Assertions**: visual snapshot 일치 (Playwright `toHaveScreenshot`)

#### TS-07: 야외/저시력 시뮬레이션 (선택, P1+)

- **Given** Chromium 색맹/저대비 시뮬레이션 + 큰 글씨 모드
- **When** 핵심 액션 흐름 5종 수행
- **Then** 가독성/조작 모두 가능
- **Assertions**: axe 위반 0, 시각 회귀 임계값 통과

#### TS-08: 안전점검 체크리스트 (P1 시니어 자력)

- **Given** worker 안전점검 진입
- **When** 28px+ 체크박스 4~6항목 1탭씩 체크 → 제출
- **Then** 1분 이내 완료, 모든 항목 기록됨

#### TS-09: company 모바일 결재 1탭

- **Given** company 사장(P3) 모바일 + 결재 대기 1건
- **When** 푸시 알림 → 카드 1탭 → "승인"
- **Then** 30초 이내 처리, 카드형 보고서 미리보기 가독성

#### TS-10: 오프라인 회복력 (PWA SW)

- **Given** worker 오프라인 상태(차고지/지하)
- **When** 출근 체크인 시도
- **Then** SW 큐에 저장 + 온라인 복귀 시 자동 동기화 + 배지/토스트 알림

## 10. 실행 우선순위 (Plan 문서로 전파)

### 10.1 의도된 실행 순서

```
[Wave 0: 기반]
   1. 디자인 토큰 체계 정의 (색/폰트/간격/터치/모션) — AAA 사전 검증
   2. AppShell 컴포넌트 골격 (헤더/하단 5탭/FAB/Sheet)
   3. viewport 잠금 + safe-area + status bar 색상

[Wave 1: 통합]
   4. 4-role 라우트 모두 AppShell 내부로 이주
   5. 단일 로그인 + role 자동 라우팅
   6. 로그아웃 동선 재설계 (사용자 보고 결함 우선 해결)

[Wave 2: 분기]
   7. RBAC 메뉴 정의 + RAPID '최적경로계산' 분기
   8. 페이지별 컴포넌트 토큰 적용 (text-slate-200 같은 잔재 제거)

[Wave 3: 셸 보강]
   9. PWA 셸 (스플래시, manifest 강화, 홈화면 추가 onboarding 카드)
   10. 야외/야간 가독성 옵션 토글 (P1)

[Wave 4: 검증]
   11. axe + visual regression CI 게이트 강제
   12. 베타 위탁사 사용성 테스트 (SUS 측정)
   13. RFP 자산화
```

> **부분 패치 거부 정책**: Wave 0/1이 완료되기 전에는 단일 페이지 패치 PR을 받지 않는다 (운영 핫이슈 hotfix는 별도 브랜치 + 본 이니셔티브 토큰 흡수 시 즉시 재흡수).
