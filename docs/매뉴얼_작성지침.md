# 사용자 매뉴얼 작성 지침서

> CleanERP 사용자 매뉴얼(`/manual`)을 수정·확장할 때 따라야 할 가이드. 이 문서를 읽고 따라 하시면 디자인을 깨뜨리지 않고 카피·챕터·역할을 추가하실 수 있습니다.
> **핵심 원칙: 디자인은 코드가, 카피는 콘텐츠 파일이 책임진다.**

**대상 독자**: 매뉴얼을 유지보수할 운영자·개발자·콘텐츠 작성자
**최종 갱신**: 2026-05-02
**관련 문서**: [소개서_작성지침.md](./소개서_작성지침.md)

---

## 1. 매뉴얼이란

CleanERP 사용자 매뉴얼은 **3 역할(근로자·회사관리자·지자체관리자)** 각각을 위한 단계별 사용 안내서입니다. 웹과 PDF로 동시 제공되며, 화면 목업으로 실제 사용 흐름을 시각화합니다.

### 1.1 라우트 구조

| URL | 대상 | 챕터 수 | 사이드바 색 | PDF 파일명 |
|---|---|:---:|---|---|
| `/manual` | 모든 사용자 (랜딩) | — | — | — |
| `/manual/worker` | 운전원·수거원·기동반 | 13 | — (모바일) | `근로자_사용설명서.pdf` |
| `/manual/contractor` | 대표·팀장·안전관리자 | 13 | navy `#1e3a5f` | `회사관리자_사용설명서.pdf` |
| `/manual/muni` | 시·군·구 환경과 직원 | 12 | blue `#1d4ed8` | `지자체관리자_사용설명서.pdf` |

### 1.2 핵심 원칙: 배려 (Caring)

매뉴얼의 모든 결정은 **다음 한 가지 원칙**에 따라야 합니다:

> 폰을 처음 다루시는 50대+ 환경공무직도 막힘없이 따라올 수 있어야 한다.

이 원칙이 의미하는 것:

- 본문 글자 18px+, 헤드라인 32px+
- 빨강 위협 X, 부드러운 앰버 톤 (WarnBox)
- 모든 절차는 ①②③ 번호 + 화면 목업 시각화
- 격식체(~습니다·~하세요), 친근체(~해요) 사용 금지
- "이럴 때는 이렇게" (TipBox) 미리 짚어주기
- 기술 용어 직접 사용 X (예: "JWT 토큰" → "로그인 정보")

### 2.1 무엇을 만지면 되나?

| 작업 | 만지는 파일 | 영향 범위 |
|---|---|---|
| 카피 수정 (오탈자·문장 다듬기) | `_content/{role}.tsx` | 해당 챕터만 |
| 화면 목업 변경 (실제 UI 변경 반영) | `_content/{role}.tsx` (JSX 부분) | 해당 step만 |
| 새 챕터 추가 | `_content/{role}.tsx` (배열에 객체 추가) | 해당 매뉴얼만 |
| 새 역할 추가 | `_config.ts` + `_content/{newRole}.tsx` + `{newRole}/page.tsx` | 새 라우트 |
| 디자인 변경 (색·간격·폰트) | `manual.css` | **전체** |
| 새 mock primitive 추가 | `_components/` + `manual.css` | 전체 |

**가장 흔한 작업은 카피 수정 (95%)** — 콘텐츠 파일만 보시면 됩니다.

## 3. 카피 수정 (가장 흔한 작업)

### 3.1 워크플로우

1. `app/manual/_content/worker.tsx` (또는 contractor/muni) 열기
2. 수정할 챕터 찾기 — `num: '03'` 등으로 검색
3. 텍스트만 수정
4. 저장 → 브라우저에서 새로고침 (dev server 자동 재컴파일)

### 3.2 ChapterData 스키마

각 챕터는 다음 3종 중 하나입니다:

#### A. `kind: 'welcome'` — 환영합니다 챕터

```tsx
{
  kind: 'welcome',
  num: '01',                                          // 챕터 번호 (string)
  title: '환영합니다',                                // 챕터 제목
  lead: '한 챕터의 1줄 요약',                          // 도입부 (lead paragraph)
  intro: '더 긴 도입 문단 (선택)',
  hero: <ScreenShot ...>...</ScreenShot>,             // 큰 화면 미리보기 (선택)
  canDo: [
    { title: '핵심 기능 1', body: '설명 한 문장' },
    { title: '핵심 기능 2', body: '설명 한 문장' },
    // 5~6개 권장
  ],
}
```

#### B. `kind: 'standard'` — 일반 챕터 (단계별 절차)

```tsx
{
  kind: 'standard',
  num: '03',
  title: '출근 도장 찍기',
  lead: '이 챕터의 1줄 요약',
  steps: [
    {
      title: '1단계 제목 — 동사형',
      body: '본문 — 1~2 문장. 너무 길게 X.',
      screenshot: <ScreenShot ...>...</ScreenShot>,    // 선택, 핵심 단계만
    },
    // 3~5개 권장
  ],
  tip: { title: '제목', body: '본문' },               // 선택, 1챕터당 0~1개
  warn: { title: '제목', body: '본문' },              // 선택, 1챕터당 0~1개
  nextHref: '#04',                                    // 다음 챕터 anchor (선택)
  nextDesc: '다음 단계 안내 한 문장',                  // 선택
}
```

#### C. `kind: 'faq'` — 자주 묻는 질문

```tsx
{
  kind: 'faq',
  num: '13',
  title: '자주 묻는 질문',
  lead: '이 챕터의 도입 한 문장',
  faqs: [
    { q: '질문 한 문장 (물음표 포함)', a: '답변 1~3 문장' },
    // 8~12개 권장
  ],
}
```

### 3.3 카피 톤 가이드

#### ✅ 권장 패턴

| 상황 | 카피 |
|---|---|
| 동작 안내 | "출근하기 버튼을 눌러 주세요" |
| 안 풀릴 때 | "잘 안 되시면 마당으로 잠시 나오시면 빨리 잡힙니다" |
| 주의 사항 | "다른 사람과 절대 공유하지 마세요" |
| 다음 안내 | "익숙해지셨다면 다음 단계도 해보세요" |
| 결과 확인 | "등록 완료 메시지가 표시됩니다" |
| 옵션 사항 | "(선택) 홈 화면에 추가" |

#### ❌ 피할 패턴

| 안티패턴 | 이유 | 대신 |
|---|---|---|
| "~해요" | 친근체는 공식 매뉴얼 톤과 안 맞음 | "~합니다·~하세요" |
| "JWT 토큰을..." | 기술 용어 노출 | "로그인 정보를..." |
| "에러가 발생합니다" | 부정적 표현 | "잘 안 되시면 ~" |
| "**금지**" | 위협조 | "주의 부탁드립니다" |
| "삭제됩니다 ⚠️" | 빨강 톤 | WarnBox + 부드러운 어조 |

### 3.4 매뉴얼 ↔ 실 UI 일치 체크

카피 수정 시 반드시 **실제 화면과 일치하는지** 확인:

- 메뉴 위치가 실제와 같은가? (예: "더보기 탭" vs "홈 그리드 카드")
- 버튼 라벨이 실제와 같은가?
- 탭 이름이 실제와 같은가?

이전에 발견된 불일치 사례 (워커 매뉴얼):
- Ch.04 휴가: "더보기 탭" → 실제는 **홈 화면 카드**
- Ch.09 민원: 단일 화면 → 실제는 **"내 민원 / 신규 등록" 두 탭**
- Ch.12 추천경로: "더보기 메뉴" → 실제는 **홈 그리드 카드** (기동반 only)

→ 의심되면 [실제 코드](../app/worker/) 직접 확인.

## 4. 새 챕터 추가

### 4.1 단계

1. `_content/{role}.tsx` 열기
2. `WORKER_CHAPTERS` (또는 `CONTRACTOR_CHAPTERS`/`MUNI_CHAPTERS`) 배열에서 추가할 위치 결정
3. 새 ChapterData 객체 작성 — 위 스키마 참고
4. `num` 값 부여: 기존 챕터 번호와 충돌 X (예: '02-1' 같은 보조 번호도 가능)
5. 앞 챕터의 `nextHref: '#03'`이 새 챕터를 가리키도록 조정
6. 저장 → `/manual/{role}`에서 자동 렌더 확인

> Chapter 컴포넌트가 `kind`에 따라 자동으로 렌더하므로 page.tsx는 손대지 않습니다.

### 4.2 챕터 번호 규칙

- `'01'`, `'02'`, ... 두 자리 zero-pad
- 슬라이드 본문에서는 `CHAPTER 01` 형식으로 자동 표시
- HTML id 도 `id="01"`로 자동 부여 → `#01` 앵커 링크 가능

## 5. 화면 목업 작성

### 5.1 모바일 화면 — `<ScreenShot>` (워커 매뉴얼)

```tsx
import ScreenShot from '../_components/ScreenShot';
import ButtonMock from '../_components/ButtonMock';
import StatusChipMock from '../_components/StatusChipMock';
import FormRowMock from '../_components/FormRowMock';

<ScreenShot
  appBar={{ title: '출퇴근', role: 'worker', showBack: true }}
  activeTab="clock"   // 'home' | 'clock' | 'complaint' | 'safety' | 'perf'
  caption="GPS 준비 완료 후 초록 출근하기 버튼이 활성화됩니다."
>
  <FormRowMock label="현재 시각" value="07:03" />
  <FormRowMock label="GPS 상태" value={<StatusChipMock label="준비 완료 ✓" tone="success" />} />
  <ButtonMock label="출근하기" variant="success" highlighted />
</ScreenShot>
```

### 5.2 데스크톱 화면 — `<DesktopShot>` (회사관리자·지자체관리자 매뉴얼)

```tsx
import DesktopShot from '../_components/DesktopShot';
import KpiCardMock from '../_components/KpiCardMock';
import TableMock from '../_components/TableMock';

<DesktopShot
  url="wci.helpbiz.kr/dashboard"
  active="메인 대시보드"           // 사이드바에서 활성화할 메뉴 라벨 (정확히 일치)
  variant="admin"                  // 'admin' (CONTRACTOR) | 'muni' (지자체)
  caption="대시보드 4개 KPI."
>
  <div className="mock-h2">메인 대시보드</div>
  <div className="mock-kpi-grid">
    <KpiCardMock label="출근현황" value="26 / 30" tone="success" />
    <KpiCardMock label="미처리 민원" value="7" tone="warn" />
  </div>
</DesktopShot>
```

### 5.3 mock primitive 레퍼런스

| 컴포넌트 | 용도 | 핵심 props |
|---|---|---|
| `ButtonMock` | CTA 버튼 | `label`, `variant` (primary/success/danger/secondary/disabled), `highlighted` |
| `StatusChipMock` | 상태 배지 | `label`, `tone` (info/warn/success/danger/neutral) |
| `FormRowMock` | 입력/정보 1행 | `label`, `value`/`placeholder`, `type` (value/input/header) |
| `ListItemMock` | 리스트 카드 | `title`, `sub`, `right`, `highlighted` |
| `KpiCardMock` | KPI 카드 | `label`, `value`, `unit`, `sub`, `tone` |
| `TableMock` | 데이터 테이블 | `headers[]`, `rows[]` (cells + highlighted) |
| `MapPlaceholder` | 지도 영역 | `pinX`, `pinY`, `label`, `height` |

### 5.4 `highlighted` 활용 (강조 단서)

매뉴얼 step에서 "여기를 누르세요"라고 가리킬 때:

```tsx
<ButtonMock label="출근하기" variant="success" highlighted />     // 노란 점선 outline
<ListItemMock title="휴가 신청" highlighted />                     // 노란 강조
<TableMock rows={[{ cells: [...], highlighted: true }]} />          // 빨간 배경+border
```

→ 사용자 시선이 즉시 강조 위치로 가도록.

## 7. PDF 출력

### 7.1 사용자 워크플로우

1. 매뉴얼 라우트 진입 (`/manual/worker` 등)
2. 우측 상단 [PDF 저장] 버튼 클릭
3. 브라우저 인쇄 다이얼로그에서 "PDF로 저장" 선택
4. A4 portrait, 컬러 보존 옵션 활성화 권장

### 7.2 인쇄 검증 체크리스트

새 챕터·새 목업 추가 후 인쇄 미리보기로 확인:

- [ ] 챕터가 페이지 중간에서 끊기지 않음 (`page-break-inside: avoid`)
- [ ] 화면 목업의 색상 보존됨 (`print-color-adjust: exact`)
- [ ] 글자 크기가 작아지지 않음 (`@media print` 적용)
- [ ] 사이드바 색상 (admin navy / muni blue) 정확
- [ ] 강조 outline (`highlighted`) 보존됨

문제 시 [manual.css](../app/manual/manual.css) `@media print` 섹션 점검.

## 8. 자주 하는 실수

| 실수 | 결과 | 해결 |
|---|---|---|
| `_content/worker.ts` 직접 수정 | 파일 없음 (`.tsx`로 이동됨) | `worker.tsx` 수정 |
| ChapterData 객체에 `kind` 누락 | 타입 에러 | `'welcome'` / `'standard'` / `'faq'` 명시 |
| `screenshot`에 `<ScreenShot>` 외 다른 JSX | 디자인 깨짐 | 항상 ScreenShot/DesktopShot wrapper 사용 |
| `DesktopShot` `active` 라벨 오타 | 사이드바 active 안 됨 | SidebarMock의 메뉴 라벨과 정확히 일치 (예: "메인 대시보드") |
| 친근체 사용 | 매뉴얼 톤 깨짐 | "~해요" → "~합니다" 일괄 치환 |
| 빨간색 강조 | 위협 톤 | `tone="warn"` (앰버) 또는 `tone="danger"` 신중히 |
| 카피만 수정하고 화면 목업 안 바꿈 | 매뉴얼-실 UI 불일치 | step.body 변경 시 screenshot 재검토 |

## 9. 사후 동기화 (시스템 변경 반영)

CleanERP가 업데이트되면 매뉴얼 동기화가 필요할 수 있습니다.

### 9.1 점검 주기

- **분기 1회 정기 점검 권장**
- 새 기능 출시 시 즉시 (출시 전 매뉴얼 업데이트가 더 안전)
- 사용자 문의 누적 시 — 같은 질문 3건 이상 들어오면 매뉴얼 보강

### 9.2 동기화 체크리스트

각 매뉴얼별로:

- [ ] 메뉴 라벨이 실 콘솔과 일치
- [ ] 버튼 라벨·색상이 실 UI와 일치
- [ ] 탭 구조 동일
- [ ] 새 기능이 매뉴얼에 누락된 것 없음
- [ ] 폐지된 기능이 매뉴얼에 남아있지 않음
- [ ] FAQ가 최근 사용자 질문 반영

### 9.3 동기화 가이드

발견 즉시 수정하시거나, [docs/매뉴얼_변경이력.md] (없으면 생성)에 누적해 다음 업데이트 시 일괄 처리.
