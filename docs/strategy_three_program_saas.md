## 1. 현황 분석 — 두 프로그램이 만들어낸 자산

### 1.1 wci.helpbiz.kr (CleanERP)

**성격**: 운영 데이터 수집 레이어

| 구분 | 내용 |
|------|------|
| 완성도 | Phase 1~24 완료 (v0.1.0-alpha) |
| 핵심 자산 | 근태·출퇴근·휴가·차량운행일지·폐기물 처리실적·민원 등 **12개 모듈** |
| 테넌트 구조 | 지자체 → 위탁업체 → 작업자 3계층 멀티테넌트 |
| 사용자 수 | SUPER / MUNI_ADMIN / CONTRACTOR_ADMIN / INTERNAL_ADMIN / WORKER 5 Role |
| 원가 관련 | `CostCalculation` 모델(스키마만) + 월별 근무시간 자동집계 + 연료사용량·주행거리 수집 |
| 미구현 | 원가계산 API 전체 없음, 급여·임금 데이터 없음, 차량 구입가·감가상각 없음 |

**wci가 이미 수집 중인 원가 기초 데이터**:
```
인건비 기초  →  월별 근무시간(야간·연장·휴일 구분) + 직책·직급
차량비 기초  →  주행거리(시작~종료km) + 연료사용량(L) + 정비비용
실적 기초    →  폐기물 처리량(kg) + 자원순환 반입량(톤) + 운행횟수
```

### 1.2 gis.helpbiz.kr (Smart Waste Dashboard)

**성격**: 분석·예측·최적화 레이어

| 구분 | 내용 |
|------|------|
| 완성도 | 서울시 빅데이터 경진대회 출품 수준 (연구 프로토타입) |
| 핵심 자산 | GPS 181만 포인트 실증 + TSP 46.21% 거리절감 + Ridge R²=0.85 |
| 기술 스택 | Python · Streamlit · Folium · OR-Tools · scikit-learn |
| Ridge 모델 | 4개 변수(인구밀도·인력밀도·차량밀도·예산) → 1인당 폐기물 발생량 예측 |
| 원가 관련 | 거시 변수 기반 원가 추정(예측) 가능. 그러나 실제 지출 내역 미보유 |
| Gap | 실제 원가 항목(임금·감가상각·연료단가)은 포함 안 됨 |

**gis의 원가 관련 Gap 분석**:

| 원가 항목 | 현재 커버리지 | Gap |
|---------|------------|-----|
| 인건비 세부 | 인력밀도로 간접추정 | 기본급·상여금·수당 분리 불가 |
| 차량 감가상각 | 차량 수만 반영 | 구입가·연식·내용연수 없음 |
| 지역별 임금 차등 | 인구밀도로 근사 | 지자체 공시 기준표 미반영 |
| 연료 단가 변동 | 미포함 | 월별 단가 연동 없음 |
| 폐기물 종류별 차등 | 생활폐기물만 | 건설·대형·사업장 분리 불가 |

│                 │                   │                      │
│ • 작업자 앱(PWA)│ • 월별 원가 산출   │ • 경로 최적화(TSP)   │
│ • 근태 관리     │ • 입찰 원가 견적   │ • 비용 예측(Ridge)   │
│ • 차량 운행일지 │ • 사후정산 자동화  │ • 지역별 비교분석    │
│ • 폐기물 실적   │ • 계약 단가 협상   │ • AI 입찰 스카우트   │
│ • 민원 관리     │ • 손익 분석        │ • CO2 절감 시뮬      │
│                 │                   │                      │
│  데이터 생산    │  데이터 가공·정산  │  데이터 분석·예측    │
└────────────────┴───────────────────┴──────────────────────┘
         ↓ 운영데이터 공급          ↑ 실적 피드백
         └──────────────────────────┘
```

**데이터 흐름**:
```
wci (일별 실적 수집)
    → 원가산정 (월별 원가 계산·정산서 생성)
        → gis (누적 실측데이터로 예측 모델 고도화)
            → wci (최적 경로·배차 개선)
```

## 3. 원가산정 프로그램 설계

### 3.1 정의

> 위탁업체가 지자체에 청구하거나, 지자체가 예산을 편성할 때 사용하는 **공식 원가 계산·정산 도구**.  
> 법적 근거: 환경부 고시 "생활폐기물 수집·운반 대행계약을 위한 원가계산 산정방법에 관한 규정"

### 3.2 원가산정 구성 항목 (환경부 기준)

| 대분류 | 세부항목 | 데이터 출처 | 산정방식 |
|--------|---------|-----------|---------|
| **직접 인건비** | 기본급 | 급여 마스터 (신규) | 월 기본급 × 인원 |
| | 상여금 | 급여 마스터 | 기본급 × 요율(최대 400%) |
| | 제수당 | wci 근태 집계 | 야간·연장·휴일 시간 × 가산율 |
| | 퇴직급여충당금 | 급여 마스터 | 기본급 × 1/12 |
| | 4대보험 | 급여 마스터 | 기본급 × 요율 |
| **차량비** | 감가상각비 | 차량 마스터 (확장) | 구입가 ÷ 내용연수(6년) |
| | 수리·수선비 | wci 정비이력 | 실적 또는 추정치 |
| | 보험료 | 차량 마스터 | 차량별 연간 보험료 |
| **유류비** | 연료비 | wci 운행일지 + 단가 | 사용량(L) × 월별단가 |
| | 윤활유 | 차량 마스터 | 추정 단가 |
| **간접비** | 관리비 | 설정값 | 직접비 합계 × 요율 |
| | 제세금 | 설정값 | 고정값 입력 |

### 3.3 원가산정 프로그램 기능 5종

#### 기능 1: 월별 원가 자동 집계
```
wci 데이터 자동 수집:
  인건비  ← 근태집계(근무시간) + 급여 마스터
  연료비  ← 운행일지(fuelUsed) × 단가설정
  수선비  ← 정비이력(maintCost)

관리자 수동 입력:
  감가상각비 (차량 구입가 등록 후 자동)
  간접비 (월별 입력)

자동 산출:
  → 톤당 단가 = 총원가 ÷ 처리량(톤)
  → 원가비율 분석 (파이차트)
```

#### 기능 2: 입찰 원가 견적서 생성
```
입력:
  - 계약 구역 (지자체 선택)
  - 계획 인원 수 + 차량 수
  - 계약 기간

gis 모델 연동:
  - Ridge 예측 → 예상 발생량
  - TSP 최적화 → 예상 주행거리

자동 산출:
  - 인건비 견적 (인원 × 직급별 기준급)
  - 차량비 견적 (차량 수 × 차종별 감가상각)
  - 유류비 견적 (예상주행거리 × 연비 × 단가)
  - 간접비 (직접비 × 설정요율)

출력:
  → 표준 원가계산서 (HWP/PDF)
  → 계약 단가 제안서
```

#### 기능 3: 사후원가 정산 자동화
```
계약 기간 종료 후:
  wci 실적 데이터 불러오기 (월별 근태·운행·처리실적)
  실제 원가 재계산 → 계획 대비 차이 분석
  증빙 문서 자동 수집 (영수증·운행일지 PDF)
  
출력:
  → 사후원가 정산보고서 (Excel 4시트)
  → 증빙 목록 (문서번호·금액·내역)
```

#### 기능 4: 손익 분석 대시보드
```
계약 수익  vs  실제 원가
  → 월별 손익 그래프
  → 톤당 단가 추이 (계획 vs 실적)
  → 원가 항목별 비율 분석
  → 이상 발생 시 알림 (원가율 XX% 초과)
```

#### 기능 5: 지자체 예산 편성 지원
```
지자체(MUNI_ADMIN) 전용:
  - 용역 발주 전 예상 원가 산출
  - 전년도 실적 기반 다음 연도 예산 추정
  - gis의 Ridge 모델로 지역 특성 반영한 단가 산출
  - 인근 지자체 벤치마크 비교
```

### 3.4 원가산정 기술 스택

| 항목 | 결정 | 근거 |
|------|------|------|
| 프레임워크 | Next.js 14 (wci와 동일) | 테넌트 구조·인증 공유 |
| DB | wci PostgreSQL 공유 | 추가 스키마 확장 |
| 문서 출력 | python-docx / WeasyPrint | HWP 출력 필요 시 HWPX 스킬 활용 |
| gis 연동 | REST API 또는 DB 직접 | Ridge 예측값 호출 |
| 배포 | wci와 동일 Docker Compose | 서브도메인 분리(cost.helpbiz.kr) |

## 4. SaaS 통합 아키텍처

### 4.1 도메인 구조

```
helpbiz.kr (랜딩·마케팅)
├── wci.helpbiz.kr     → CleanERP (운영 ERP)
├── cost.helpbiz.kr    → 원가산정 (신규)
└── gis.helpbiz.kr     → Smart Waste (분석·대시보드)
```

### 4.2 인증 통합 전략

**단기 (지금)**: 각 프로그램 독립 인증, 동일 계정 테이블 공유
```
PostgreSQL: 사용자 테이블 1개 → wci + cost가 공유
gis: 별도 Basic Auth (연구용 유지)
```

**중기**: SSO (Single Sign-On) 구현
```
JWT 토큰 1개로 wci ↔ cost 자동 로그인
gis는 wci 토큰 검증 후 접근 허용
```

**장기**: 지자체별 커스텀 도메인
```
gangnam.helpbiz.kr → 강남구 전용 (wci + cost + gis 통합)
```

### 4.3 데이터 공유 전략

```
공유 스키마 (wci DB):
  ├── 사용자·권한 (User, Contractor, Municipality)
  ├── 운영 데이터 (VehicleLog, Attendance, WasteRecord)
  └── 원가 데이터 (CostCalculation + 확장 테이블)

gis 연동:
  ├── 매월 1회 배치: wci 실적 → gis 학습 데이터 갱신
  └── 온디맨드 API: 원가산정 → gis Ridge 예측값 요청
```

### 4.4 요금제 통합 (SaaS 과금)

| 플랜 | 월정액 | 포함 모듈 | 대상 |
|------|--------|---------|------|
| STARTER | 50만원 | wci만 | 소규모 위탁업체 |
| STANDARD | 150만원 | wci + cost | 중형 업체 |
| PRO | 350만원 | wci + cost + gis 기본 | 대형 업체·컨소시엄 |
| ENTERPRISE | 협상 | 전체 + 커스터마이징 | 광역 지자체 |

현재 wci 견적 자동화 명세(docs/specs/02_견적자동화_명세서.md)의 요금 구조와 연동 가능.

## 6. 핵심 의사결정 요약

### Q1. 원가산정을 wci에 합칠까 vs 분리할까?

**권장: 분리 (cost.helpbiz.kr)**

| 항목 | wci 통합 | 분리 (권장) |
|------|---------|-----------|
| 개발 속도 | 빠름 | 약간 느림 |
| 사용자 혼란 | 높음 (작업자 앱에 회계 UI) | 낮음 |
| 판매 유연성 | wci 없이 cost만 판매 불가 | cost 단독 판매 가능 |
| 지자체 직접 판매 | 어려움 | 가능 (예산 편성 도구로) |

### Q2. gis를 언제 SaaS로 전환할까?

**권장: 경진대회 수상 후 (2026년 Q3~Q4)**

- 현재 gis는 연구 프로토타입 (Streamlit, NAS 배포)
- 서울시장상 레퍼런스 확보 후 → Next.js 리라이트 + AWS 이관
- wci/cost와 동일한 테넌트 구조로 편입

### Q3. 원가산정의 첫 고객은 누구인가?

**우선순위**: 기존 wci 위탁업체 → 지자체 순

1. wci를 이미 쓰는 위탁업체: 데이터가 이미 있으니 바로 원가 산출 가능
2. 입찰 준비 중인 신규 위탁업체: 견적 기능으로 진입
3. 지자체: 예산 편성 지원으로 진입 (늦은 단계)

## 7. 요약 — 한 장 정리

```
현재 자산:
  wci  →  운영 데이터 12모듈 완성, 원가 스키마만 있고 API 없음
  gis  →  예측 모델(R²=0.85), 실측 데이터 없음

빠진 연결고리:
  원가산정  =  wci의 실적 데이터 + gis의 예측 모델을 엮는 정산 허브

3개 프로그램의 분업:
  wci     →  "오늘 뭘 얼마나 했나" (운영)
