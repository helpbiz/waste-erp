## 2. 핵심 개념 3가지 (초보 개발자용)

### 2.1 멀티테넌시(Multi-tenancy) — "한 건물에 여러 회사 입주"

- 하나의 CleanERP 인스턴스 안에 **여러 위탁업체**가 함께 존재하지만, **데이터는 절대 섞이지 않음**
- 비유: 같은 빌딩(서버) 안에 A회사·B회사 사무실이 있지만, A회사 서류는 B회사가 못 봄
- **구현 핵심**: 모든 데이터에 `contractorId`(어느 회사 소속) + `municipalityId`(어느 지자체 관할) 태그가 자동 부착되어야 함
- 검증: 분기 1회 cross-tenant 누출 audit 필수

### 2.2 RBAC (Role-Based Access Control) — "직책에 따라 보이는 게 다름"

- 같은 화면에 들어가도 **로그인한 사람의 Role**에 따라 보이는 메뉴·버튼·데이터가 다름
- SUPER_ADMIN이 일일이 권한 부여 X → **Role(직책)에 권한을 묶어두고 사람에게 Role을 부여**

### 2.3 계층 구조 (Hierarchy) — "위가 아래를 포함"

```
SUPER_ADMIN (운영사 = 헬프비즈)
   └── MUNI_ADMIN (지자체 = 강남구청)
         └── CONTRACTOR_ADMIN (위탁업체 = 강남청소(주))
               └── INTERNAL_ADMIN (회사 내부 팀장)
                     └── WORKER (운전원·수거원·기동반·청소원)
```

## 3. Role 5단계 정의

현재 [prisma/schema.prisma](../../prisma/schema.prisma) `enum Role`에 정의된 5단계입니다.

| Role | 한글명 | 대표 사용자 | 핵심 역할 | 관리 범위 | 모바일 사용 |
|---|---|---|---|---|:---:|
| **SUPER_ADMIN** | 시스템관리자 | 헬프비즈 운영팀 | 플랫폼 전체 운영 | 모든 회사·모든 지자체 | △ (콘솔 위주) |
| **MUNI_ADMIN** | 지자체관리자 | 강남구청 환경과 직원 | **모니터링만** (조회 위주) | 자기 관할 위탁업체들 | △ |
| **CONTRACTOR_ADMIN** | 회사관리자 | 위탁업체 대표·임원 | 회사 운영 총괄 | 자기 회사 1곳 | ✅ |
| **INTERNAL_ADMIN** | 일반관리자 | 위탁업체 팀장·실장·안전관리자 | 결재·배차·실적·안전 관리 | 자기 회사 1곳 (실무) | ✅ |
| **WORKER** | 일반근로자 | 운전원·수거원·기동반·청소원 | 모바일 앱 사용 | 본인 데이터만 | ✅ (주력) |

### 3.1 권한 위임 흐름 (정상 케이스)

```
SUPER_ADMIN
   │
   │ ① 인스턴스 셋업 + 위탁업체 등록 + CONTRACTOR_ADMIN 계정 1개 발급
   ▼
CONTRACTOR_ADMIN (위탁업체 대표)
   │
   │ ② 자기 회사 INTERNAL_ADMIN(팀장) 계정 + WORKER(직원) 계정 직접 등록
   ▼
INTERNAL_ADMIN
   │
   │ ③ 일상 운영 (결재·배차·민원 배정)
   ▼
WORKER (모바일 앱 사용)
```

### 3.2 MUNI_ADMIN은 별도 트랙

- 회사 소속이 없는 **외부 감독자** — SUPER_ADMIN이 직접 발급
- "보여주고 싶은 것만 보여주는" 권한 매트릭스로 제어 (현재 슈퍼관리자 콘솔 Tab 2)
- 데이터 수정 권한 없음 — **조회·다운로드만**

## 5. 신규 고객 서비스 개설 절차 (개발자 관점)

[서비스_개설_절차.md](../../서비스_개설_절차.md) 14일 일정을 **개발자/운영자 작업 관점**으로 4단계로 압축.

### 5.1 단계별 책임 분담

```
[Day 0~5]  계약 단계 (영업·법무 영역) ─ 개발자 무관
   ↓
[Day 6~7]  ★ 시스템 셋업 ─ 운영자 작업 (본 문서 핵심)
   │
   ├─ 1. 지자체 등록 (이미 226개 시군구 사전 시드 → 거의 skip)
   ├─ 2. 권한 매트릭스 설정 (지자체에 무엇을 보여줄지)
   ├─ 3. 위탁업체(Contractor) 레코드 생성
   ├─ 4. CONTRACTOR_ADMIN 계정 1개 발급 (임시 PW)
   └─ 5. 차고지 좌표 등록 (지오코딩)
   ↓
[Day 8~10] 데이터 마이그레이션 (CSV import)
   ├─ 직원 명단 일괄 import (`scripts/import-employees.ts`)
   ├─ 차량 명단 일괄 import (`scripts/import-vehicles.ts`)
   └─ 결재 라인 정의
   ↓
[Day 11~14] 교육·시범·정식 개시 ─ 영업·교육 영역
```

### 5.2 운영자가 ★단계에서 실제로 하는 일

**모두 `/super-admin` 콘솔에서 클릭으로 처리** — SQL 직접 입력 X.

| # | 화면 | 액션 | 소요 |
|---|---|---|:---:|
| 1 | `/super-admin` Tab 1 | 지자체 ACTIVE 확인 (이미 시드됨) | 30초 |
| 2 | `/super-admin` Tab 2 | 권한 매트릭스 편집 → 프리셋 선택 | 1분 |
| 3 | `/super-admin` Tab 5 | 위탁업체 신규 등록 → 차고지 입력 | 3분 |
| 4 | `/users` | CONTRACTOR_ADMIN 계정 생성 → 임시 PW 발급 | 2분 |
| 5 | (메일) | 접속 정보 안전 채널 발송 | 1분 |

→ **현재 총 ~10분, 마법사 도입 시 ~3분**.

### 5.3 마이그레이션 단계 검증 체크리스트

- [ ] 등록된 직원 수 = CSV 행 수 (skip 제외)
- [ ] WORKER 계정으로 모바일 로그인 테스트 (랜덤 1명)
- [ ] CONTRACTOR_ADMIN 계정으로 직원 목록 조회 정상
- [ ] **다른 위탁업체 직원 데이터 안 보임 (cross-tenant)** ← 가장 중요
- [ ] 차량 모두 ACTIVE 상태
- [ ] 결재 라인 정상 흐름 (테스트 휴가 신청 1건)

## 7. "어렵게 느껴지는" 이유 분석 + 단순화 5처방

### 7.1 어렵게 느껴지는 4가지 원인

1. **개념이 5겹**: 지자체 → 위탁업체 → 부서 → 직원 → 권한... 입체적으로 그려야 함
2. **순서 의존**: 지자체부터 만들어야 위탁업체를 묶을 수 있음. 거꾸로 못 함
3. **다중 권한 모호**: 한 사람이 여러 Role을 가질 수 있는지(현재 1인 1Role) 헷갈림
4. **MUNI_ADMIN이 특수**: 회사 소속이 없는 외부인인데 데이터를 봐야 함

### 7.2 단순화 처방 5종

#### 처방 1: **신규 업체 개설 마법사** ⭐⭐ 가장 추천

현재는 4~5개 탭을 왔다갔다 해야 하지만, 이를 **하나의 위저드(Wizard)로 통합**:

```
[1단계] 회사 정보 입력
        - 사업자번호 (자동 진위 확인 가능)
        - 회사명·대표자·연락처
        - 차고지 주소 (자동 지오코딩)
   ↓
[2단계] 관할 지자체 선택
        - 226개 사전 등록된 지자체 중 검색
        - 다중 선택 가능 (광역업체 대응)
   ↓
[3단계] 권한 정책 선택
        - 프리셋 3종 중 1개: [표준] [모니터링만] [전체 공개]
        - 또는 [세부 편집] 옵션
   ↓
[4단계] CONTRACTOR_ADMIN 1명 등록
        - 이름·이메일·전화
        - 임시 PW 자동 생성 (12자 랜덤)
   ↓
[5단계] CSV로 직원 일괄 등록 (선택, 나중에 가능)
        - 직원 CSV·차량 CSV 드래그&드롭
        - 또는 [건너뛰기] → CONTRACTOR_ADMIN이 직접 등록
   ↓
[완료]
        - 접속 URL + 임시 PW + QR코드 PDF 자동 생성
        - 메일 발송 (CONTRACTOR_ADMIN 대표·운영팀 BCC)
        - 셋업 체크리스트(5.3) 자동 검증
```

**효과**: 클릭 ~20+ → ~5, 운영자 학습 시간 1주 → 1일.

#### 처방 2: **권한 프리셋 3종**

지자체 권한 매트릭스를 매번 클릭으로 설정하지 말고:

| 프리셋 | 허용 화면 | 다운로드 | 적용 케이스 |
|---|---|:---:|---|
| **A. 표준** | 대시보드+민원+보고서+안전 | ✅ | 90% — 기본값 |
| **B. 모니터링 전용** | 대시보드만 | ❌ | 소극적 지자체·시범 단계 |
| **C. 전체 공개** | 모든 화면 | ✅ (bulk 포함) | 광역단체·전수 검토 케이스 |

→ 운영자가 망설일 일 없음. "잘 모르면 A".

#### 처방 3: **역할 트리 시각화**

슈퍼관리자 콘솔 첫 화면(또는 별도 탭)에 트리 다이어그램:

```
헬프비즈 (운영사) — SUPER_ADMIN: 1명
├─ 강남구청 (지자체) — MUNI_ADMIN: 김민원 (1명)
│   └─ 강남청소(주) — CONTRACTOR_ADMIN: 박대표 (1명)
│       ├─ 운전부 (팀) — INTERNAL_ADMIN: 이팀장 (1명)
│       │   └─ WORKER 12명 ▼

```

→ 누가 어디 소속인지 1초 만에 보임. 영업 시연에도 강력.

#### 처방 4: **위임 가능 작업 체크리스트** (§4 매트릭스 인쇄용)

## 9. 작업 우선순위 (1인 운영 기준)

### Phase 1 — 즉시 (1주, ~12~15h)

**목적**: 신규 개설 부담 즉시 경감. DB 변경 없음.

|---|---|:---:|---|
| P1-1 | 권한 프리셋 3종 정의 + 매트릭스 탭에 "프리셋 적용" 버튼 | 2h | Tab 2 |
| P1-2 | 본 문서(§4) 권한 매트릭스 → 인쇄용 PDF 생성 스크립트 | 1h | 문서 |
| P1-3 | 신규 개설 마법사 5단계 모달 (UI만, API는 기존 재사용) | 8h | Tab 5 |
| P1-4 | 마법사 완료 후 셋업 체크리스트(5.3) 자동 검증 표시 | 2h | Tab 5 |
| P1-5 | 마법사 완료 시 임시 PW + QR PDF 자동 생성·메일 | 2h | API + 메일 |

### Phase 2 — 정식 도입 후 (2주, ~16~20h)

**목적**: SUPER_ADMIN 누락 메뉴 4종 보강 + 트리 시각화.

| # | 작업 | 공수 |
|---|---|:---:|
| P2-1 | SUPER_ADMIN 전용 "사용자 관리(전체)" 메뉴 신설 | 6h |
| P2-2 | SUPER_ADMIN 전용 "시스템 모니터링" 대시보드 | 6h |
| P2-3 | SUPER_ADMIN 전용 "감사 로그 뷰어" (필터·검색) | 4h |
| P2-4 | 역할 트리 시각화 (D3 또는 react-flow, dynamic import) | 4h |

### Phase 3 — Nice-to-Have (별도 1~2주)

| # | 작업 | 공수 | 비고 |
|---|---|:---:|---|

| P3-4 | 회사 soft-delete 30일 보관 (Q4-B 채택 시) | 4h | DB + 정리 cron |

## 11. 위험 요소 및 mitigation

| # | 위험 | 영향 | 대응 |
|---|---|:---:|---|
| 1 | 마법사 단계 도중 실패 시 부분 데이터 잔존 | High | 트랜잭션으로 묶거나, 단계별 idempotent + "재개" 기능 |
| 2 | CONTRACTOR_ADMIN이 INTERNAL_ADMIN을 너무 많이 발급 (남용) | Medium | 회사별 INTERNAL_ADMIN 상한 (요금제별) |
| 3 | 권한 프리셋 변경 시 기존 매트릭스 덮어쓰기 사고 | Medium | 적용 전 diff 미리보기 + 확인 모달 |
| 4 | MUNI_ADMIN이 여러 지자체 관리(Q3-B) → 데이터 격리 복잡도↑ | Medium | 우선 Phase 3로 분리, 영업 단계 결정 후 진입 |
| 5 | 신규 메뉴(P2-1~4) 도입으로 SUPER_ADMIN 콘솔 사이드바 길어짐 | Low | 카테고리 그룹핑 ("운영" / "모니터링" / "청구") |
| 6 | 트리 시각화 라이브러리(react-flow ~200KB) 도입으로 번들 증가 | Low | dynamic import — 트리 탭 진입 시점에만 로드 |
