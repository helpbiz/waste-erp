## 1. 시스템 개요 + 아키텍처

### 1.1 한 줄 정의
**지자체 위탁 환경미화 운영을 디지털화한 멀티테넌트 ERP** — 시민·작업자·관리자·지자체·슈퍼관리자가 같은 시스템을 다른 권한으로 사용.

### 1.2 도메인 구성

```
[시민]              [작업자]            [관리자]        [지자체 담당자]    [슈퍼관리자]
   │                   │                    │                │              │
   ▼                   ▼                    ▼                ▼              ▼
/citizen          /worker           /complaints, /users  /reports     /super-admin
(인증 X)        (모바일 PWA)       /attendance, /safety  (R/O 가능)   (전 권한)
   │                   │                    │                │              │
   └───────────────────┴────────────┬───────┴────────────────┴──────────────┘
                                    ▼
                       [Next.js 14 App Router (3001)]
                                    │
                          JWT(jose) + RBAC 미들웨어
                                    │
                                    ▼
                          [PostgreSQL 16 (5434)]
                                    │
                          ┌─────────┴─────────┐
                          ▼                   ▼
                    [OSRM 도로 라우팅]   [OSM Nominatim 지오코딩]
                    router.project-osrm  nominatim.openstreetmap
                          ▼
                    [빼기 앱 (외부) — 대형폐기물 cron import]
```

| 점검 항목 | 위치 | 비정상 기준 | 조치 |
|---|---|---|---|

| 빼기 cron 결과 | `/bulky-waste` 실행 이력 | "FAILED" 표시 | 9.5장 참조 |
| 휴가 잔여 0 | `/users` 연월차관리 탭 | 잔여 0 + 신청 대기 | 결재 진행 요청 |

```bash
# 최근 1시간 ERROR 로그
docker logs --since 1h cleanerp-app 2>&1 | grep -iE "error|fatal" | tail -20

# 5xx 응답 발생 여부
docker logs --since 1h cleanerp-app 2>&1 | grep -E "HTTP/[0-9.]+ (5[0-9]{2})" | wc -l
```

## 3. 주간 점검

> **소요**: 약 30분
> **수행**: 매주 월요일 오전

### 3.1 백업 정상성 확인

```bash
# 최근 7일치 백업 파일 존재 확인 (S3 또는 로컬)
ls -lh /opt/backups/cleanerp-*.sql.gz | tail -7

# 가장 최근 백업의 무결성 검사 (압축 검증)
gunzip -t /opt/backups/cleanerp-$(date +%Y%m%d -d yesterday).sql.gz
echo "exit code: $?"   # 0이면 OK

# 분기 1회 — 실제 복구 리허설 (별도 DB로)
# → 16.4장 참조
```

### 3.2 audit_logs 검토

```sql
-- 최근 1주 SUPER_ADMIN 활동
SELECT created_at, action, resource_type, resource_id, ip_address
FROM audit_logs
WHERE actor_role = 'SUPER_ADMIN'
  AND created_at >= NOW() - INTERVAL '7 days'
ORDER BY created_at DESC
LIMIT 50;

-- 비정상 로그인 실패 패턴 (같은 IP, 5분 내 5회+)
SELECT ip_address, COUNT(*) as fail_count
FROM audit_logs
WHERE action = 'LOGIN_FAIL'
  AND created_at >= NOW() - INTERVAL '7 days'
GROUP BY ip_address, date_trunc('minute', created_at)
HAVING COUNT(*) >= 5
ORDER BY fail_count DESC;
```

### 3.3 사용량 통계

```sql
-- 지자체별 활성 사용자 수
SELECT m.region, m.name, COUNT(DISTINCT u.id) as users
FROM municipalities m
LEFT JOIN users u ON u.municipality_id = m.id AND u.status = 'ACTIVE'
GROUP BY m.id, m.region, m.name
ORDER BY users DESC;

-- 모듈별 주간 활성도
SELECT
  date_trunc('day', created_at) as day,
  COUNT(*) FILTER (WHERE action LIKE 'COMPLAINT_%') as complaints,
  COUNT(*) FILTER (WHERE action LIKE 'LEAVE_%') as leaves,
  COUNT(*) FILTER (WHERE action LIKE 'ATTENDANCE_%') as attendance
FROM audit_logs
WHERE created_at >= NOW() - INTERVAL '7 days'
GROUP BY day
ORDER BY day DESC;
```

## 6. 데이터베이스 관리

### 6.1 접속 방법

```bash
# Docker 컨테이너 내부 (운영)
docker exec -it cleanerp-postgres psql -U cleanerp -d cleanerp_prod

# 호스트에서 직접
PGPASSWORD='<.env.prod의 POSTGRES_PASSWORD>' \
  psql -h localhost -p 5434 -U cleanerp -d cleanerp_prod

# Prisma Studio (GUI)
DATABASE_URL=postgresql://cleanerp:...@localhost:5434/cleanerp_prod \
  npx prisma studio
```

### 6.2 스키마 변경 절차

```bash
# 1. prisma/schema.prisma 편집
# 2. 변경 검증 (drift 점검)
DATABASE_URL=... npx prisma db push --accept-data-loss=false

# 3. 운영 적용 (반드시 백업 후!)
DATABASE_URL=... pg_dump -Fc > backup-pre-migration-$(date +%Y%m%d-%H%M).dump

DATABASE_URL=... npx prisma db push   # apply
DATABASE_URL=... npx prisma generate  # regen client

# 4. Docker 재빌드 (Prisma client 변경 반영)
docker compose -f docker-compose.prod.yml --env-file .env.prod up -d --build app
```

> ⚠️ **컬럼 삭제·타입 변경**은 별도 마이그레이션 스크립트로 단계적 진행. `db push --accept-data-loss`는 절대 사용 금지.

### 6.3 자주 사용하는 진단 쿼리

```sql
-- 테이블별 행 수 + 크기
SELECT
  schemaname || '.' || relname AS table,
  n_live_tup AS rows,
  pg_size_pretty(pg_total_relation_size(schemaname||'.'||relname)) AS size
FROM pg_stat_user_tables
ORDER BY pg_total_relation_size(schemaname||'.'||relname) DESC
LIMIT 20;

-- 슬로우 쿼리 (pg_stat_statements 활성화 필요)
SELECT query, calls, mean_exec_time, total_exec_time
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 20;

-- 인덱스 미사용 검출
SELECT schemaname, relname, indexrelname, idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND schemaname = 'public'
ORDER BY relname;

-- 활성 연결 수
SELECT state, COUNT(*)
FROM pg_stat_activity
WHERE datname = 'cleanerp_prod'
GROUP BY state;
```

### 6.4 마이그레이션 충돌 시

```bash
  --from-schema-datamodel prisma/schema.prisma \
  --to-schema-datasource prisma/schema.prisma \
  --script

# 결과 SQL 검토 후 수동 적용 → 다시 db push
```

## 12. 신규 지자체 온보딩 절차

> **소요**: 약 30분 (지자체 1곳 기준)
> **수행자**: SUPER_ADMIN

### 12.1 단계별 절차

```
1단계: 지자체 등록
  → /super-admin → "지자체 관리" 탭
  → "+ 지자체 신규 등록"
  → 행정코드 5자리 + 이름 + 광역단체 + 상태=ACTIVE
       (전국 226개 시군구는 이미 시드되어 있음 — 검색만 하면 됨)

2단계: 지자체 권한 정책 설정
  → "지자체 권한 매트릭스" 탭
  → 해당 지자체 → "권한 편집"
  → 허용 화면 / 허용 보고서 / 다운로드 권한 체크
  → 저장

3단계: 산하 위탁업체 등록
  → /users → 위탁업체 추가
  → 사업자등록번호 + 대표자 + 차고지 주소 (지오코딩 자동)

4단계: MUNI_ADMIN 계정 발급
  → /users → 신규 사용자
  → role: MUNI_ADMIN
  → municipality: 신규 등록한 지자체
  → 임시 비밀번호 발급 → 첫 로그인 시 강제 변경

5단계: 인수인계 자료 전달
  → 사용자_설명서.md 출력 또는 PDF
  → 접속 URL + 임시 계정
  → 첫 로그인 시 /consent 동의 안내
```

## 16. 백업 + 재해 복구 (DR)

### 16.1 일일 자동 백업 cron

```bash
# /etc/cron.d/cleanerp-backup
0 3 * * * root /opt/scripts/cleanerp-backup.sh

# /opt/scripts/cleanerp-backup.sh
#!/bin/bash
set -e
DATE=$(date +%Y%m%d)
BACKUP_DIR=/opt/backups
mkdir -p $BACKUP_DIR

# pg_dump
docker exec cleanerp-postgres \
  pg_dump -U cleanerp cleanerp_prod -Fc | gzip > $BACKUP_DIR/cleanerp-$DATE.sql.gz

# S3 업로드 (옵션)
aws s3 cp $BACKUP_DIR/cleanerp-$DATE.sql.gz \
  s3://cleanerp-backup/db/ --storage-class STANDARD_IA

# 90일 이전 로컬 삭제
find $BACKUP_DIR -name 'cleanerp-*.sql.gz' -mtime +90 -delete

echo "✓ Backup $DATE done"
```

### 16.2 백업 무결성 확인

```bash
# 압축 검증
gunzip -t /opt/backups/cleanerp-20260426.sql.gz

# 첫 100줄 미리보기
gunzip -c /opt/backups/cleanerp-20260426.sql.gz | head -100
```

### 16.3 복구 절차

```bash
# Step 1 — 운영 DB 백업 (덮어쓰기 전 안전망)
docker exec cleanerp-postgres pg_dump -U cleanerp cleanerp_prod -Fc \
  > /tmp/before-restore-$(date +%Y%m%d-%H%M).dump

# Step 2 — 운영 앱 정지
docker compose -f docker-compose.prod.yml --env-file .env.prod stop app

# Step 3 — DB 초기화 + 복원
docker exec -i cleanerp-postgres psql -U cleanerp -d postgres -c \
  "DROP DATABASE IF EXISTS cleanerp_prod; CREATE DATABASE cleanerp_prod OWNER cleanerp;"

gunzip -c /opt/backups/cleanerp-<날짜>.sql.gz | \
  docker exec -i cleanerp-postgres pg_restore -U cleanerp -d cleanerp_prod

# Step 4 — 앱 재기동 + 검증
docker compose -f docker-compose.prod.yml --env-file .env.prod start app
curl http://localhost:3001/api/health
```

### 16.4 복구 리허설 (분기)

```bash

  -p 5499:5432 postgres:16-alpine

gunzip -c /opt/backups/cleanerp-$(date +%Y%m%d -d yesterday).sql.gz | \
  docker exec -i pg-test pg_restore -U postgres -d postgres

# 데이터 정합성 확인

# 정리

```

### 16.5 RPO/RTO 목표

| 지표 | 목표 |
|---|---|
| RPO (Recovery Point Objective) | **24시간** (일일 백업) |
| RTO (Recovery Time Objective) | **2시간** (앱 재기동 + DB 복원) |

> 더 엄격한 목표 필요 시: PITR (Point-In-Time Recovery) 활성화 → WAL 아카이빙 + 시간 단위 복구

```sql
-- 1. audit_log 5년 도달 데이터 (보존 정책 준비)
SELECT MIN(created_at) FROM audit_logs;
-- 5년 도달 시 archive 테이블로 이전 정책 수립

-- 2. PII 접근 audit
SELECT actor_role, COUNT(*) AS access_count
FROM audit_logs
WHERE action LIKE '%USER_VIEW%'
  AND created_at >= NOW() - INTERVAL '90 days'
GROUP BY actor_role;

-- 3. 동의 미진행 사용자 (이상치)
SELECT id, username FROM users
WHERE privacy_consent_at IS NULL AND status = 'ACTIVE';
-- 첫 로그인 후 미동의 사용자는 정상 (사용 차단됨)
```

## 🎯 운영 철학

1. **변경은 작게, 자주, 가역적으로** — 큰 배포 1회보다 작은 배포 100회
2. **자동화하라** — 사람이 매번 하는 일은 스크립트로
3. **로그는 거짓말하지 않는다** — 추측 전에 로그를 본다
4. **백업 없는 변경은 없다** — 모든 위험 작업 전 dump 필수
5. **혼자 결정하지 말라** — 모호하면 묻고, 위험하면 동료 리뷰
6. **사용자 시점에서 사고하라** — 우리가 만든 것은 사용자의 도구
7. **개인정보는 최소 수집·최단 보관** — 법은 최저 기준일 뿐
8. **에스컬레이션은 부끄러움이 아니다** — 빠른 도움 요청이 빠른 복구
